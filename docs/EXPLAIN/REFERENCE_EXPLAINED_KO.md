# 참고문헌 깊게 이해하기

이 문서는 M-RAG 논문이 인용한 35편의 참고문헌을 각각 설명한다. 각 논문이 **무엇을 발견했는지**, **M-RAG가 그것을 어떻게 구체적으로 활용하는지**, **이 논문 없이는 M-RAG의 어느 부분이 만들어질 수 없었는지**를 중심으로 설명한다.

---

본문의 숫자와 짧은 컨텍스트 조각은 각 논문의 아이디어를 설명하기 위한 예시다. 실제 M-RAG 실험 수치와 최종 표는 `backend/evaluation/results/` 산출물을 기준으로 본다.

## RAG 전체 설계 근거

### [1] Gao et al. — RAG Survey (2023)

**무엇을 발견했는가**

RAG 시스템은 크게 3가지 단계로 나뉜다는 걸 정리한 조사 논문이다.
- **Retrieval(검색)**: 관련 문서를 찾는 단계
- **Augmentation(증강)**: 찾은 문서를 모델 입력에 추가하는 단계
- **Generation(생성)**: 문서를 참고해 답변하는 단계

또한 RAG의 진화 단계를 Naive RAG → Advanced RAG → Modular RAG 3단계로 분류했다. Modular RAG는 각 단계를 독립적인 모듈로 분리해 교체하거나 조합할 수 있는 설계다.

**M-RAG에서 어떻게 쓰이는가**

M-RAG의 이름(Modular RAG)과 6개 파이프라인(A–F) 설계가 이 논문의 Modular RAG 분류 체계에서 직접 출발했다. "어떤 기능을 독립 모듈로 만들 것인가"를 결정할 때 이 논문의 분류를 기준으로 삼았다.

---

### [20] Lewis et al. — RAG Original (2020)

**무엇을 발견했는가**

"문서를 검색한 뒤 그 문서를 근거로 언어 모델이 답변한다"는 아이디어를 처음 논문으로 발표한 연구다. 이전에는 언어 모델이 훈련 데이터만 보고 답변했다. Lewis et al.은 질문이 들어올 때마다 외부 문서를 검색해 함께 입력으로 넣으면, 모델이 훈련 중에 없었던 사실도 답할 수 있다는 걸 보였다.

**M-RAG에서 어떻게 쓰이는가**

M-RAG 전체 흐름이 이 논문의 RAG 패러다임을 따른다. "질문 → 검색 → 문서+질문을 모델에 입력 → 생성"이라는 기본 구조 자체가 이 논문에서 왔다.

---

### [13] Wang et al. — Best Practices in RAG (2024)

**무엇을 발견했는가**

"RAG를 만들 때 어떤 설계 선택이 실제로 성능을 높이는가"를 50가지 이상의 실험으로 비교했다. 예를 들어:
- 청크 크기가 256 토큰일 때와 512 토큰일 때 검색 성능이 얼마나 다른가
- Reranker를 검색 후에 추가하면 얼마나 개선되는가
- Context Compressor가 실제로 생성 품질을 높이는가

**M-RAG에서 어떻게 쓰이는가**

M-RAG의 청크 크기 설정(약 500 토큰), Reranker 위치(검색 바로 다음), Context Compressor 적용 여부 등 구체적인 구현 선택의 근거로 사용했다. "왜 500 토큰인가"에 대한 답이 이 논문에 있다.

## 검색 모듈 근거

### [2] Chen et al. — BGE-M3 (2024)

**무엇을 발견했는가**

기존 임베딩 모델은 주로 영어만 잘 표현했다. 한국어 문장을 영어 문장과 같은 벡터 공간에서 비교하면 정확도가 낮았다. BGE-M3는 100개 이상 언어를 같은 벡터 공간에서 학습해, 언어가 달라도 의미가 같으면 벡터가 가까워지도록 만든 모델이다.

구체적으로:
```
[영어 문장] "The model subtracts parametric knowledge from output logits."
[한국어 문장] "모델이 출력 로짓에서 파라메트릭 지식을 뺀다."

[BGE-M3 변환]
  영어 벡터: [0.021, -0.143, 0.892, ...]
  한국어 벡터: [0.019, -0.141, 0.889, ...]
  코사인 유사도: 0.97 (거의 같음)

[일반 영어 임베딩 모델]
  영어 벡터: [0.021, -0.143, 0.892, ...]
  한국어 벡터: [-0.312, 0.741, -0.112, ...]
  코사인 유사도: 0.31 (거의 다름)
```

**M-RAG에서 어떻게 쓰이는가**

임베딩 모듈(`backend/modules/embedder.py`)에서 BGE-M3를 사용한다. 한국어로 "CAD의 alpha 값이 뭐야?"라고 물어도 영어 논문 청크를 정확하게 찾을 수 있는 핵심 이유가 이 모델 덕분이다. BGE-M3 없이 영어 전용 임베딩을 쓰면 한국어 질문으로 영어 논문을 검색하는 능력이 크게 떨어진다.

### [22] Robertson et al. — BM25

**무엇을 발견했는가**

BM25는 1994년에 제안된 키워드 기반 검색 알고리즘이다. 검색어가 문서 안에 얼마나 자주 나오는지(TF: Term Frequency)와 전체 문서 중 얼마나 적은 수의 문서에서 나오는지(IDF: Inverse Document Frequency)를 함께 계산해 점수를 매긴다.

예시: "alpha" 라는 단어가
- 문서 1개에서 10번 나옴 → TF가 높아 점수 상승
- 전체 100개 청크 중 2개에만 나옴 → IDF가 높아 점수 상승 (희귀한 단어일수록 가치 높음)

**M-RAG에서 어떻게 쓰이는가**

하이브리드 검색(`backend/modules/hybrid_retriever.py`)에서 Dense 검색과 나란히 실행된다. Dense 검색(BGE-M3)은 의미적 유사도를 잡고, BM25는 정확한 키워드 매칭을 잡는다.

"CAD alpha 값"처럼 특정 기술 용어가 들어간 질문에서 BM25가 특히 강하다. Dense 검색만으로는 "alpha"라는 단어가 없지만 의미상 비슷한 청크를 가져올 수 있다. BM25는 "alpha"라는 단어가 실제로 있는 청크를 우선 선택한다.

---

### [23] Cormack et al. — RRF: Reciprocal Rank Fusion (2009)

**무엇을 발견했는가**

두 검색 시스템이 각각 순위를 매긴 결과가 있을 때, 이 두 순위를 하나로 합치는 공식이다.

```
RRF 점수 = 1/(k + Dense순위) + 1/(k + BM25순위)
(k = 60, 일반적으로 고정)
```

예시:
| 청크 | Dense 순위 | BM25 순위 | RRF 점수 |
|---|---|---|---|
| 청크 A | 1위 | 3위 | 1/61 + 1/63 = 0.0321 |
| 청크 B | 5위 | 1위 | 1/65 + 1/61 = 0.0315 |
| 청크 C | 2위 | 2위 | 1/62 + 1/62 = 0.0323 ← 최종 1위 |

양쪽에서 모두 높은 순위를 받은 청크 C가 최종적으로 가장 높은 점수를 받는다.

**M-RAG에서 어떻게 쓰이는가**

Dense 검색 결과와 BM25 결과를 합칠 때 이 공식을 그대로 사용한다(`backend/modules/hybrid_retriever.py`). RRF 없이 두 리스트를 합치면, 점수 단위가 달라서 직접 더하거나 비교할 수 없다(Dense 점수는 코사인 유사도 0.0~1.0, BM25는 0~50 범위). RRF는 순위만 사용하므로 단위 문제가 없다.

---

### [14] Khattab & Zaharia — ColBERT (2020), [15] ColBERT v2

**무엇을 발견했는가**

일반 Dense 검색은 질문 전체를 하나의 벡터로 압축하고, 문서 전체도 하나의 벡터로 압축해서 비교한다. 정보가 하나의 점으로 압축되는 과정에서 세부 정보가 손실된다.

ColBERT는 다르다. 질문의 각 단어를 벡터로 만들고, 문서의 각 단어도 벡터로 만들어서, 질문 단어 하나하나와 문서 단어 하나하나를 비교(Late Interaction)한다. 더 정밀하지만 계산량이 많다.

**M-RAG에서 어떻게 쓰이는가**

M-RAG의 Reranker 단계는 ColBERT의 "단어 단위 정밀 비교" 아이디어를 참조해 설계됐다. 실제 Reranker는 Cross-Encoder를 사용하지만, ColBERT처럼 "하나의 벡터로 압축하지 말고 더 정밀하게 비교하자"는 철학이 Reranker 도입의 이론적 배경이다.

---

## 쿼리 확장과 라우팅 근거

### [6] Gao et al. — HyDE: Hypothetical Document Embeddings (2022)

**무엇을 발견했는가**

"질문 문장"과 "답변 문장"은 표현 방식이 다르다는 문제를 발견했다. 논문은 "We propose...", "Our method achieves..."처럼 서술 형식으로 쓰여 있다. 질문은 "어떻게 해?", "뭐야?"처럼 의문 형식이다. 벡터 공간에서 이 두 형식은 의미가 같아도 벡터가 다를 수 있다.

해결책: 질문을 보고 먼저 가상의 답변 문장을 만들어, 그 가상 답변으로 검색한다. 가상 답변은 논문 문체와 비슷하기 때문에 실제 논문 문장과 벡터가 더 가깝다.

```
[원래 질문]
  "CAD에서 alpha 값을 높이면 어떻게 되나요?"
  벡터: [의문 패턴, 미래 추측 패턴, ...]

[HyDE가 만든 가상 답변]
  "CAD에서 alpha 값을 높이면 컨텍스트 의존도가 증가하고
   파라메트릭 지식 개입이 감소한다."
  벡터: [서술 패턴, CAD 개념, alpha, 컨텍스트, ...]

[실제 논문 청크]
  "A higher alpha value increases the weight on context-aware
   logits and suppresses parametric knowledge."
  벡터: [서술 패턴, CAD 개념, alpha, 컨텍스트, ...]

→ 가상 답변과 실제 논문 청크의 벡터가 질문 벡터보다 훨씬 가깝다
```

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/query_expander.py`에 HyDE를 구현했다. 경로 A(일반 QA)에서 검색 전에 항상 HyDE를 거친다. 이 덕분에 질문 형식의 표현 차이에도 불구하고 논문의 관련 부분을 정확히 찾는다.

### [7] Asai et al. — Self-RAG (2023)

**무엇을 발견했는가**

RAG 시스템이 항상 검색을 해야 하는 건 아니다. "2+2는 얼마야?"처럼 모델이 이미 알고 있는 질문에는 검색이 오히려 방해가 된다. Self-RAG는 모델이 "지금 검색이 필요한가?"를 스스로 판단하게 하는 방법을 제안했다.

또한 검색 결과가 실제로 질문에 유용한지도 스스로 평가(ISREL: Is Relevant?)하고, 답변이 근거에 충실한지도 스스로 확인(ISSUP: Is Supported?)하는 자기 반성 구조를 도입했다.

**M-RAG에서 어떻게 쓰이는가**

Query Router(`backend/modules/query_router.py`) 설계의 이론적 배경이다. M-RAG는 질문 유형에 따라 다른 경로(A~F)를 선택하는데, "질문 유형에 따라 다른 처리 경로를 선택한다"는 아이디어가 Self-RAG에서 왔다.

---

### [8] Yan et al. — CRAG: Corrective RAG (2024)

**무엇을 발견했는가**

검색 결과가 질문과 관련이 없을 때, 그냥 포기하지 말고 다시 시도하거나 웹 검색으로 보완하자는 아이디어다. CRAG는 검색 결과를 평가해 "CORRECT", "INCORRECT", "AMBIGUOUS" 세 등급으로 나누고, 낮은 등급이면 재검색을 시도한다.

**M-RAG에서 어떻게 쓰이는가**

검색 품질 보장 로직의 참고 문헌이다. M-RAG도 Context Compressor 이후 컨텍스트가 비어 있거나 품질이 낮으면 검색 파라미터를 조정한다는 아이디어에 CRAG의 개념이 반영됐다.

---

### [26] Shi et al. — RAG-Fusion (2024)

**무엇을 발견했는가**

하나의 질문을 여러 가지 표현으로 바꿔서(예: 4~5가지 다른 문장으로) 각각 검색하면, 같은 질문으로 한 번만 검색할 때보다 더 다양한 관련 문서를 찾을 수 있다. 예를 들어:

```
원래 질문: "CAD의 alpha 값이 성능에 어떤 영향을 미쳐?"
재작성 1: "CAD에서 alpha 파라미터의 역할은?"
재작성 2: "alpha 값 변화에 따른 Faithfulness 변화"
재작성 3: "CAD 하이퍼파라미터 alpha 최적값"

→ 4가지 다른 쿼리로 4번 검색 → 결과를 RRF로 합산
```

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/query_expander.py`에서 다중 쿼리 생성을 지원한다. HyDE(가상 답변 하나 생성) 외에 쿼리 재작성(여러 표현 생성)도 사용한다. 검색 커버리지를 높이는 데 기여한다.

---

## 문서 구조와 요약 근거

### [5] Sarthi et al. — RAPTOR (2024)

**무엇을 발견했는가**

논문 전체를 요약해야 하는 질문에 일반 청크 검색은 한계가 있다. 30개의 청크 중 5개를 뽑아봤자 전체 흐름의 20%도 파악하기 어렵다. RAPTOR는 다음 방식으로 해결했다:

```
[기존 RAG]
  30개 청크 → 질문과 유사한 5개 선택 → 조각 수준 답변

[RAPTOR]
  30개 청크
    → 5개씩 묶어 요약 (청크 레벨 요약 6개)
    → 2개씩 묶어 요약 (섹션 레벨 요약 3개)
    → 전체 요약 1개

  "전체 요약"에 대한 질문 → 전체 요약 청크가 검색됨
  "Method에 대한 질문" → Method 섹션 요약이 검색됨
  "세부 수치 질문" → 원래 청크가 검색됨
```

계층적 구조 덕분에 질문 수준에 따라 적절한 깊이의 내용을 반환할 수 있다.

**M-RAG에서 어떻게 쓰이는가**

경로 E(전체 요약, `backend/pipelines/pipeline_e_summary.py`)가 RAPTOR 방식을 활용한다. "이 논문 전체를 요약해줘"라는 질문에 청크 수준이 아니라 전체 요약 수준의 내용을 반환할 수 있는 이유가 RAPTOR의 계층 구조 덕분이다.

---

### [24] Chen et al. — Dense-X Retrieval (2023), [28] Meta-Chunking

**무엇을 발견했는가**

Dense-X Retrieval은 텍스트를 단락 단위가 아니라 "명제(proposition)" 단위로 나누면 검색 정확도가 높아진다는 걸 발견했다. 명제는 "사실 하나"를 담은 문장이다.

```
[단락 단위 청킹]
  "CAD는 2023년에 제안됐으며, alpha=0.5가 최적이었다.
   실험은 3개 데이터셋에서 진행됐으며, TriviaQA, NQ, HotpotQA가 사용됐다."
  (2개 사실이 섞여 있음 → 검색 시 정확도 저하)

[명제 단위 청킹]
  명제 1: "CAD는 2023년에 제안됐다."
  명제 2: "CAD의 최적 alpha는 0.5다."
  명제 3: "실험은 TriviaQA, NQ, HotpotQA에서 진행됐다."
  (각 청크가 정확히 하나의 사실만 담음)
```

Meta-Chunking은 여러 청킹 전략을 혼합해 사용하면 더 좋다는 걸 보였다.

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/chunker.py`의 청킹 전략 설계에서 참고했다. 고정 크기 청킹 외에 섹션 단위, 명제 단위 옵션을 제공하는 근거가 이 논문들이다.

### [11] Jiang et al. — LLMLingua (2023), [12] LongLLMLingua, [19] Xu et al. — RECOMP

**무엇을 발견했는가**

검색으로 가져온 청크 전체를 모델에 넣으면 비효율적이다:
- 청크의 상당 부분은 질문과 관련이 없다
- 언어 모델의 입력 한도를 초과할 수 있다
- 관련 없는 내용이 많으면 답변 품질이 떨어진다

LLMLingua는 청크에서 중요도가 낮은 토큰을 제거해 80% 이상 압축하면서도 핵심 정보는 유지할 수 있다는 걸 보였다. RECOMP는 청크를 그대로 쓰지 않고 질문에 맞게 요약해 컨텍스트로 만드는 방법을 제안했다.

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/context_compressor.py`의 설계 근거다. 검색 결과 10개 청크를 그대로 모델에 넣는 대신, 핵심 문장만 남겨 압축한다. 이 덕분에 답변 품질이 높아지고 생성 속도도 빨라진다.

## 생성 제어 근거

### [3] Shi et al. — CAD: Context-Aware Decoding (2023)

**무엇을 발견했는가**

언어 모델은 문서를 보지 않고도 "그럴듯한 답변"을 만들 수 있다. 그 이유는 훈련 중에 수많은 텍스트를 읽어 "비슷한 질문에는 보통 이런 답이 나온다"는 패턴을 기억하기 때문이다. 이를 파라메트릭 지식(parametric knowledge)이라 부른다.

문제는 이 기억이 실제 문서 내용과 다를 때다. 예를 들어:
- 논문에 쓰인 값: "실험 결과 Faithfulness = 87.3%"
- 모델의 기억: "이런 실험에서는 보통 90%쯤 나온다"
→ 문서를 봐도 모델이 90%라고 답할 수 있다

CAD는 이를 수식으로 해결했다:
```
최종 로짓 = 로짓(문서+질문) - alpha × 로짓(질문만)
```
"문서 없이 나온 답변 경향"을 직접 빼버린다.

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/cad_decoder.py`에 직접 구현했다. M-RAG의 모든 경로(A~F)의 답변 생성 단계에 적용된다. 논문 기준 실험에서는 alpha 고정값(0.0~1.0 sweep)을 사용한다. 이것이 M-RAG의 핵심 기여 중 하나(C2: CAD+SCD 조합)의 절반이다.

**M-RAG가 원 논문과 다르게 변형한 것**

원 CAD 논문은 단일 언어(영어) 환경을 가정했다. M-RAG는 이를 다국어(한국어 질문 + 영어 논문) 환경에 적용하고, SCD와 조합해 언어 일관성 문제도 함께 해결했다는 점이 확장이다.

---

### [4] Li et al. — Contrastive Decoding (2022)

**무엇을 발견했는가**

큰 언어 모델(Expert)과 작은 언어 모델(Amateur)을 동시에 실행하고, 두 모델의 출력 차이를 활용하면 큰 모델만 쓸 때보다 답변 품질이 높아진다는 걸 보였다.

```
최종 = 로짓(Expert) - beta × 로짓(Amateur)
```

작은 모델이 "그럴듯하게 잘못 말하는 패턴"을 빼냄으로써, 큰 모델이 더 정확한 답을 선택하게 만든다.

**M-RAG에서 어떻게 쓰이는가**

CAD의 이론적 기반이다. CAD의 수식(`로짓(문서+질문) - alpha × 로짓(질문만)`)은 Contrastive Decoding의 `로짓(Expert) - beta × 로짓(Amateur)` 구조에서 영감을 받아 "문서 없는 모델 = Amateur" 역할로 대입한 것이다.

---

### [34] Li et al. — Language Drift & SCD (2025)

**무엇을 발견했는가**

한국어로 질문하고 영어 논문을 검색 결과로 사용하면, 모델이 한국어로 답변하다가 영어 표현이 섞이는 "Language Drift" 현상이 발생한다는 걸 발견했다. 이 문제는 영어 컨텍스트가 모델의 언어 선택에 영향을 미치기 때문에 생긴다.

해결책으로 SCD(Selective Context-aware Decoding)를 제안했다:
```
다음 토큰 후보 중 목표 언어(한국어)가 아닌 토큰의 확률에 beta 페널티 적용
최종 확률(영어 토큰) = 원래 확률 - beta
```

**M-RAG에서 어떻게 쓰이는가**

`backend/modules/scd_decoder.py`에 직접 구현했다. CAD와 SCD를 함께 적용하는 조합이 M-RAG의 두 번째 핵심 기여(C2)다. 원 논문(2025)이 제안한 SCD를 M-RAG의 다국어 RAG 환경에 맞게 구현했다.

**M-RAG가 원 논문과 다르게 변형한 것**

원 SCD 논문은 단독으로 SCD를 제안했다. M-RAG는 CAD와 SCD를 **동시에 적용**해 (1) Faithfulness 향상(CAD)과 (2) 언어 일관성 향상(SCD)을 함께 달성하는 조합 효과를 실험으로 검증했다. 이 조합 실험이 M-RAG의 독자적인 기여다.

---

## 평가 근거

### [9] Es et al. — RAGAS (2023)

**무엇을 발견했는가**

RAG 시스템을 사람 없이 자동으로 평가할 수 있는 4가지 지표를 정의했다:

| 지표 | 측정 대상 | 측정 방법 |
|---|---|---|
| Faithfulness | 답변이 컨텍스트에 충실한가 | LLM이 SUPPORTED/PARTIAL/UNSUPPORTED 판정 |
| Answer Relevancy | 답변이 질문에 관련 있는가 | 답변에서 역으로 질문을 만들어 유사도 측정 |
| Context Precision | 검색된 컨텍스트가 유용한가 | LLM이 USEFUL/PARTIAL/NOISY 판정 |
| Context Recall | 정답 내용이 컨텍스트에 있는가 | 정답과 컨텍스트 겹침 측정 |

핵심은 "LLM을 평가자(judge)로 쓴다"는 점이다. 사람이 채점하는 대신 언어 모델이 채점한다. 이를 LLM-as-Judge라고 부른다.

**M-RAG에서 어떻게 쓰이는가**

`backend/evaluation/ragas_eval.py`에서 RAGAS의 4가지 지표를 직접 구현했다. 단, M-RAG는 RAGAS 라이브러리를 외부에서 가져와 쓰지 않고, MIDM 모델을 judge로 사용해 직접 재구현했다. RAGAS 공식 라이브러리는 GPT-4o를 judge로 사용하는데, M-RAG는 오픈소스 로컬 모델만으로 완전히 실행 가능해야 하기 때문이다.

### [18] Wei et al. — Chain-of-Thought Prompting (2022)

**무엇을 발견했는가**

복잡한 문제를 풀 때 "단계적으로 생각하는 과정을 프롬프트에 포함시키면" 언어 모델의 정확도가 크게 높아진다는 걸 발견했다.

예시:
```
[기존 프롬프트]

  모델 출력: "54" (틀림)

  모델 출력: "56" (맞음)
```

단계적 사고 과정을 적어주면 모델이 각 단계를 따라가며 더 정확히 답한다.
