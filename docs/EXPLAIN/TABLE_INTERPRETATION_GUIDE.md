# 실험표 해석 가이드

## 목적

실험표는 숫자를 채우는 것이 목적이 아니다. 어떤 모듈이 어떤 문제를 줄였는지 설명할 수 있어야 한다.

---

## Table 1 모듈 누적 ablation

보는 법

- Naive RAG는 기준선이다
- 모듈이 추가될수록 검색 품질과 답변 품질이 좋아지는지 본다
- 특정 단계에서 점수가 떨어지면 그 모듈이 항상 나쁘다는 뜻은 아니다
- 쿼리 분포, judge 품질, 컨텍스트 품질을 함께 봐야 한다

좋은 패턴

- Full System의 Faithfulness가 기준선보다 높음
- Context Precision이 검색 개선 단계에서 상승
- 한국어 질의와 영어 질의의 격차가 줄어듦

주의할 패턴

- 모든 config의 Context Precision이 동일하면 평가 또는 검색 샘플링 문제 가능성
- Faithfulness가 모두 0.95 이상이면 judge가 차이를 구분하지 못할 수 있음

---

## Table 2 CAD/SCD 조합

보는 법

- CAD only는 수치 환각이나 근거 없는 답변을 줄이는지 본다
- SCD only는 언어 이탈률을 줄이는지 본다
- CAD+SCD는 두 문제가 동시에 완화되는지 본다

주의할 점

- CAD는 답변을 더 보수적으로 만들 수 있다
- SCD는 영어 전문용어까지 억제할 수 있다
- 둘을 같이 쓰면 속도가 느려질 수 있다

---

## Table 3 alpha/beta sweep

보는 법

- alpha는 CAD 강도다
- beta는 SCD 강도다
- 값이 클수록 항상 좋은 것은 아니다

해석

- alpha가 너무 낮으면 지식 개입 억제가 약하다
- alpha가 너무 높으면 답변 다양성이 줄 수 있다
- beta가 너무 낮으면 언어 이탈 억제가 약하다
- beta가 너무 높으면 영어 기술 용어가 부자연스럽게 억제될 수 있다

---

## Table 4 Track 2 도메인 특화 비교

보는 법

- 논문 구조를 활용하는 모듈이 실제로 도움이 되는지 본다
- 섹션 필터는 Context Precision에 영향을 주는 것이 자연스럽다
- 인용 추적은 citation 질의에서 의미가 크다
- 요약 경로는 Answer Relevancy와 전체 설명 품질을 함께 봐야 한다

## 숫자가 이상할 때 점검 순서

1. 검색 결과가 실제로 달라지는지 `/api/chat/search`로 확인
2. Judge가 label을 제대로 반환하는지 `/api/chat/judge`로 확인
3. ground truth가 비어 있거나 잘못 연결되지 않았는지 확인
4. 같은 결과 JSON을 재사용하고 있지 않은지 확인
5. 전체 실험이 아닌 일부 resume 결과만 보고 있지 않은지 확인
