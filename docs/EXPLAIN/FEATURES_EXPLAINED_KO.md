# M-RAG 기능 깊게 이해하기

이 문서는 M-RAG의 각 기능이 **실제로 무슨 일을 하는지**, **이 방법을 왜 선택했는지**, **다른 방법 대신 이것을 쓰면 결과가 어떻게 달라지는지**를 구체적인 예시로 설명한다. 한 번도 RAG 시스템을 본 적 없는 사람이 읽어도 "이 기능이 왜 있는지"를 설명할 수 있을 만큼 구체적으로 쓴다.

아래의 수치, 토큰 확률, 짧은 컨텍스트 예시는 설명용이다. 실제 실험 수치와 최종 표는 `backend/evaluation/results/` 산출물을 기준으로 읽는다.

## 문서 업로드 — 파일을 검색 가능한 조각으로 바꾸기

### 실제로 무슨 일이 벌어지는가

PDF 파일을 올리면 다음 5단계가 자동으로 일어난다.

```
[사용자가 올린 파일]
  CAD_paper.pdf (15페이지 영어 논문)
          ↓
[1단계: 텍스트 추출]
  "We propose Context-Aware Decoding (CAD), which subtracts..."
  → pdfplumber가 각 페이지의 글자를 추출
  → 표, 그림 캡션, 수식은 별도 처리
          ↓
[2단계: 섹션 감지]
  "1. Introduction", "2. Method", "3. Experiments", "4. Conclusion"
  → 대문자 제목 패턴, 번호 패턴으로 경계 탐지
  → 각 청크에 섹션 태그 부착 (예: section="Method")
          ↓
[3단계: 청킹]
  전체 텍스트를 약 500 토큰 단위 조각으로 분할
  "...logits_final = logits_with_ctx - α × logits_without_ctx.
   This formulation ensures that the model's parametric knowledge..."
  → 조각 간 50 토큰 겹침 (경계 손실 방지)
          ↓
[4단계: 임베딩]
  각 조각 → BGE-M3 → 1024차원 숫자 배열
  [0.021, -0.143, 0.892, 0.015, ..., -0.331]  (1024개 숫자)
          ↓
[5단계: 벡터 저장]
  ChromaDB에 저장: 조각 텍스트 + 벡터 + 메타데이터(섹션, 페이지번호, 문서ID)
```

관련 코드: `backend/modules/chunker.py`, `backend/modules/embedder.py`, `backend/modules/vector_store.py`

### 이렇게 하지 않으면 어떻게 되는가

**PDF를 그대로 저장하고 질문이 들어오면?**
- 언어 모델은 PDF를 직접 읽지 못한다 (파일 포맷이 달라서)
- 설령 텍스트로 변환해 전부 넣어도, 15페이지 논문은 약 15,000 토큰이라 모델의 입력 한도를 초과한다
- 관련 없는 내용까지 전부 들어가므로 모델이 혼란스럽다

**조각으로 나누지 않고 전체를 검색하면?**
- 어떤 부분이 관련 있는지 알 수 없어 검색이 불가능하다
- RAG의 핵심은 "질문에 관련된 부분만 뽑아서 모델에 넣는 것"이다

### 사용자별 격리

같은 서버를 10명이 써도, 각자의 문서는 분리된 공간에 저장된다. A가 올린 논문을 B가 검색할 수 없다. 격리는 `user_id` 기반 ChromaDB 컬렉션 분리로 구현된다.

## 경로 A — 일반 QA

### 어떤 질문에 쓰이는가

특별한 의도가 없는 기본 질문.

- "BERT가 뭐야?"
- "이 논문의 결론이 뭐야?"
- "CAD의 alpha 값이 뭘 의미해?"

### 실제 입력에서 출력까지: 단계별 예시

**질문**: "CAD에서 alpha 값을 높이면 어떻게 돼?"

```
[입력] "CAD에서 alpha 값을 높이면 어떻게 돼?"
          ↓
[HyDE: 가상 답변 먼저 만들기]
  모델이 이런 가상 답변 문장을 생성:
  "CAD에서 alpha 값을 높이면 컨텍스트 의존도가 증가하고
   사전 지식의 영향이 감소한다..."
  → 이 가상 문장으로 검색 (실제 질문보다 논문 문체에 가깝다)
          ↓
[하이브리드 검색: Dense + BM25]
  Dense: "alpha 값이 클수록 모델이 문서에 더 의존..." (벡터 유사도 0.82)
  BM25:  "alpha를 0.5로 설정했을 때..." (키워드 매칭 점수 14.3)
  RRF로 합산 → 최종 후보 10개
          ↓
[Reranker: 재정렬]
  Cross-encoder가 질문-청크 쌍을 재평가
  "alpha 값이 높을수록 parametric knowledge suppression..." → 1위
  "The hyperparameter alpha controls..." → 2위
          ↓
[Context Compressor: 핵심 문장만]
  10개 청크 → 3개 핵심 문장으로 압축
  불필요한 수식 설명, 다른 파라미터 설명 제거
          ↓
[CAD+SCD 생성]
  최종 답변: "alpha 값이 높아질수록 모델이 문서 내용에 더
  충실하게 답변하고, 사전 학습에서 온 기억의 영향이 줄어든다.
  실제 답변은 제공된 문맥에 기록된 alpha 설정을 근거로 설명한다."
```

### HyDE 없이 바로 검색하면 어떻게 달라지는가

| | HyDE 사용 | HyDE 없음 |
|---|---|---|
| 검색 쿼리 | "CAD에서 alpha가 높아지면 컨텍스트 의존도가 증가..." | "CAD에서 alpha 값을 높이면 어떻게 돼?" |
| 유사도 | 논문의 실제 답변 문장과 표현 유사 | 질문 형식과 논문 서술 형식이 다름 |
| 검색 결과 | 실제 alpha 설명 섹션 탐지 | "어떻게 되는가?" 패턴과 비슷한 문장 탐지 (노이즈) |

관련 코드: `backend/pipelines/pipeline_a_simple_qa.py`, `backend/modules/query_expander.py`

## 경로 B — 섹션 특화 QA

### 어떤 질문에 쓰이는가

논문의 특정 섹션을 묻는 질문.

- "이 논문의 연구 방법이 뭐야?" → Method 섹션
- "한계점이 뭐야?" → Limitation 섹션
- "핵심 결과가 뭐야?" → Result 섹션
- "논문 요약해줘" → Abstract 섹션
- "결론이 뭐야?" → Conclusion 섹션

### 경로 A와 뭐가 다른가

경로 A는 논문 전체에서 검색한다. 경로 B는 **해당 섹션만 우선 검색**한다.

**예시: "이 논문의 연구 방법이 뭐야?"**

```
[경로 A로 처리할 경우]
  전체 논문에서 "method", "approach" 관련 청크 검색
  → Introduction의 "we propose..." 도 검색됨
  → Experiment 섹션의 "we implemented..." 도 검색됨
  → 실제 Method 섹션 내용과 관련 없는 내용이 섞임

[경로 B로 처리할 경우]
  먼저 section="Method" 태그로 필터링
  → Method 섹션 청크만 대상으로 검색
  → 정확히 방법론만 담긴 내용 반환
```

### 어떻게 어떤 섹션인지 판단하는가

질문 라우터가 질문 텍스트에서 키워드를 감지한다.

| 감지 키워드 | 매핑 섹션 |
|---|---|
| "연구 방법", "어떻게 했어", "방법론" | Method |
| "한계", "한계점", "약점" | Limitation |
| "결과", "성능", "점수" | Result |
| "결론", "마무리" | Conclusion |
| "요약", "한 줄로", "전체적으로" | Abstract |

관련 코드: `backend/pipelines/pipeline_b_section.py`, `backend/modules/query_router.py`

## 경로 C — 멀티 문서 비교

### 어떤 질문에 쓰이는가

두 개 이상의 논문/문서를 비교하거나 대조하는 질문.

- "BGE-M3와 ColBERT의 차이가 뭐야?"
- "CAD와 SCD가 각각 어떤 문제를 해결해?"
- "두 논문의 실험 설계가 어떻게 달라?"

### 실제로 어떻게 동작하는가

```
[질문] "CAD와 SCD가 각각 어떤 문제를 해결해?"
          ↓
[라우터: 비교 질문 감지]
  "와", "차이", "각각", "비교" 키워드 → 경로 C 선택
  관련 문서 식별: CAD 논문, SCD 논문
          ↓
[문서별 병렬 검색]
  CAD 논문 검색: "CAD는 어떤 문제를 해결하는가"
    → "CAD solves the problem of parametric knowledge interference..."
  SCD 논문 검색: "SCD는 어떤 문제를 해결하는가"
    → "SCD addresses language drift in multilingual generation..."
          ↓
[비교 프롬프트 조합]
  [문서1 - CAD 논문 관련 내용]
  (CAD 논문의 핵심 청크들)

  [문서2 - SCD 논문 관련 내용]
  (SCD 논문의 핵심 청크들)

  위 두 문서를 비교해 공통점과 차이점을 설명하라.
          ↓
[답변 생성]
  공통점: 두 방법 모두 언어 모델의 생성을 외부에서 제어한다.
  차이점: CAD는 문서 충실도(Faithfulness) 문제를 해결하고,
          SCD는 언어 일관성(Language Drift) 문제를 해결한다.
```

### 경로 A로 처리하면 어떻게 달라지는가

- 경로 A는 두 논문을 **하나의 검색 공간**에서 동시에 검색
- "CAD 논문에서 10개, SCD 논문에서 10개"가 아니라 전체에서 상위 10개 선택
- 한 논문이 더 많이 나올 수 있어 편향된 비교가 나올 수 있음
- 경로 C는 **각 문서에서 동일 비중으로** 내용을 가져와 균형 있는 비교 보장

관련 코드: `backend/pipelines/pipeline_c_compare.py`

## 경로 D — 인용/특허 추적

### 어떤 질문에 쓰이는가

- "이 논문이 인용한 핵심 연구는?"
- "이 논문은 어떤 선행 연구를 기반으로 해?"
- "이 특허에서 참조하는 기술이 뭐야?"

### 실제로 어떻게 동작하는가

```
[질문] "이 CAD 논문이 인용한 핵심 논문은?"
          ↓
[참고문헌 파싱]
  논문 PDF에서 References 섹션 추출
  "Lewis et al. (2020). Retrieval-Augmented Generation..."
  "Shi et al. (2023). Large Language Models Can Be..."
  → 10개 인용 항목 파싱
          ↓
[arXiv 실시간 조회]
  Lewis et al. 2020 → arXiv API → 제목, 요약, 저자, 링크 수집
  Shi et al. 2023 → arXiv API → 제목, 요약, 저자, 링크 수집
          ↓
[답변 생성]
  "이 논문이 인용한 핵심 선행 연구는 3가지입니다:
   1. RAG 원본 (Lewis et al., 2020): 검색 강화 생성을 처음 제안한 논문
   2. Contrastive Decoding (Li et al., 2022): CAD의 이론적 기반
   3. ..."
```

### 왜 본문 검색으로는 이 질문에 답할 수 없는가

논문 본문에는 "우리는 RAG(Lewis et al., 2020)의 방법을 따랐다"처럼 인용 번호나 저자 이름만 나온다. 그 논문의 실제 내용은 논문 안에 없다. 인용 논문의 내용을 알려면 외부에서 직접 가져와야 한다. 경로 D가 바로 이 역할을 한다.

관련 코드: `backend/pipelines/pipeline_d_citation.py`, `backend/modules/citation_tracker.py`, `backend/modules/patent_tracker.py`

## 경로 E — 전체 요약

### 어떤 질문에 쓰이는가

- "이 논문 전체를 요약해줘"
- "이 논문이 뭘 연구한 건지 설명해줘"
- "이 논문의 핵심 주장을 정리해줘"

### 단순 검색으로 전체 요약을 만들 수 없는 이유

일반 검색은 질문과 비슷한 **부분**을 찾는다. 그런데 "전체 요약"은 논문의 처음부터 끝까지 흐름이 필요하다.

예를 들어 청크 상위 5개를 뽑으면 이런 문제가 생긴다:
- Introduction 2개, Method 1개, Experiment 2개 → Conclusion이 빠짐
- 논문 전체의 스토리(왜 → 어떻게 → 결과 → 결론)가 연결되지 않음

### RAPTOR 방식으로 어떻게 해결하는가

```
[논문의 청크 30개 있다고 가정]

[1단계: 단락 요약]
  청크 1–5 → 요약 S1: "이 논문은 CAD를 제안한다..."
  청크 6–10 → 요약 S2: "실험은 3개 데이터셋에서..."
  청크 11–15 → 요약 S3: "결과는 기존 방법 대비 3%..."
  ... (6개 요약 생성)

[2단계: 섹션 요약]
  S1+S2 → 섹션 요약 SS1: "Method: CAD를 제안하고 구현함..."
  S3+S4 → 섹션 요약 SS2: "Experiment: 3개 데이터에서 성능 확인..."
  ... (3개 섹션 요약 생성)

[3단계: 전체 요약]
  SS1+SS2+SS3 → 논문 전체 요약 1개

[최종 답변]
  계층 구조의 요약을 조합해 논문 전체 흐름 설명
```

관련 코드: `backend/pipelines/pipeline_e_summary.py`

## 경로 F — 퀴즈/플래시카드 생성

### 어떤 질문에 쓰이는가

- "이 논문으로 퀴즈 만들어줘"
- "핵심 개념으로 플래시카드 만들어줘"
- "이 논문 내용으로 시험 문제 3개 만들어줘"

### 실제로 어떻게 만들어지는가

```
[질문] "이 논문으로 퀴즈 3개 만들어줘"
          ↓
[검색: 핵심 개념 청크 수집]
  논문에서 중요도 높은 청크 10개 선택
  (수식, 결과 수치, 핵심 정의가 있는 청크 우선)
          ↓
[퀴즈 생성 프롬프트]
  "아래 논문 내용을 바탕으로 4지선다 퀴즈를 만들어라.
   반드시 아래 내용에 있는 사실만 사용할 것.
   [논문 내용]
   CAD의 alpha 값은 0.5가 최적이었으며..."
          ↓
[출력]
  Q1. CAD 논문에서 최적 alpha 값으로 보고된 값은?
    ① 0.1  ② 0.3  ③ 0.5  ④ 1.0
    정답: ③ (근거: "alpha=0.5 achieved the best results...")

  Q2. CAD가 해결하려는 핵심 문제는?
    ① 검색 속도  ② 사전 지식 간섭  ③ 언어 혼합  ④ 청크 크기
    정답: ② (근거: "parametric knowledge interference...")
```

### 단순히 LLM에게 "퀴즈 만들어줘"라고 하면 안 되는 이유

프롬프트에 논문 내용을 포함하지 않으면, 모델이 알고 있는 내용으로 문제를 만든다. 이 경우:
- 실제 논문의 수치와 다를 수 있음 ("alpha=0.5" → 모델이 "alpha=1.0"으로 꾸밀 수 있음)
- 이 논문에 없는 일반 지식 문제가 생성됨

경로 F는 반드시 업로드된 논문에서 검색한 내용만 사용해 퀴즈를 만든다.

관련 코드: `backend/pipelines/pipeline_f_quiz.py`

## 후속 질문 자동 생성

### 왜 필요한가

논문을 처음 읽는 사람은 "다음에 뭘 물어봐야 하는지"를 모른다. 답변 뒤에 관련 질문을 2–3개 제안하면 탐색이 자연스럽게 이어진다.

### 실제 예시

```
[사용자 질문] "CAD가 뭐야?"

[답변]
  CAD는 Context-Aware Decoding의 약자로...
  (답변 내용)

[자동 생성된 후속 질문]
  ① "CAD의 alpha 값이 성능에 어떤 영향을 미쳐?"
  ② "CAD와 SCD를 함께 쓰면 어떤 차이가 나?"
  ③ "CAD가 적용되지 않으면 어떤 문제가 생겨?"
```

이 질문들은 LLM이 답변 내용을 보고 "이걸 이해했다면 자연스럽게 생길 다음 궁금증"을 예측해서 만든다.

관련 코드: `backend/modules/followup_generator.py`

## CAD 생성 제어

### 문제 상황: 모델이 "자기 기억"으로 답하는 경우

언어 모델은 훈련 중에 수많은 텍스트를 읽었다. 그 과정에서 "비슷한 실험은 보통 90% 정확도가 나온다"처럼 패턴을 기억한다.

**문제**: 논문에 "우리 실험 결과: 87.3%"라고 써 있어도, 모델이 "보통 90%야"라는 자기 기억으로 답할 수 있다.

### CAD가 어떻게 이 문제를 해결하는가

CAD는 두 번 추론한다:
1. **문서 있는 추론**: 논문 내용 + 질문 → 로짓(각 단어의 확률)
2. **문서 없는 추론**: 질문만 → 로짓

그리고 최종 로짓을 이렇게 계산한다:
```
최종 로짓 = 문서 있는 로짓 - alpha × 문서 없는 로짓
```

**구체적 예시** (alpha = 0.3):

| 다음 단어 후보 | 문서 있는 확률 | 문서 없는 확률 | 최종 (0.3 적용) |
|---|---|---|---|
| "87.3%"     | 0.70 | 0.10 | 0.70 - 0.3×0.10 = **0.67** |
| "90%"       | 0.20 | 0.60 | 0.20 - 0.3×0.60 = **0.02** |
| "88%"       | 0.10 | 0.30 | 0.10 - 0.3×0.30 = **0.01** |

**결과**: "90%"의 확률이 0.20 → 0.02로 급락. "87.3%"가 선택됨.

alpha 값이 클수록 자기 기억의 영향이 더 크게 줄어든다.

관련 코드: `backend/modules/cad_decoder.py`

## SCD 생성 제어

### 문제 상황: 한국어로 질문했는데 영어가 섞인 답변

영어 논문을 기반으로 한국어로 답변할 때, 이런 현상이 발생할 수 있다:

**Language Drift 예시**:
```
[사용자] "이 논문의 핵심 기여가 뭐야?"

[CAD만 적용, SCD 없는 답변]
  "이 논문의 핵심 기여는 We propose a novel Context-Aware Decoding
  method that significantly improves faithfulness while maintaining
  relevancy입니다."
  (영어 논문에서 가져온 표현이 그대로 섞임)

[CAD+SCD 적용 답변]
  "이 논문의 핵심 기여는 새로운 문맥 인식 디코딩 방법(CAD)을
  제안한 것입니다. 이 방법은 충실도를 높이면서 관련성도 유지합니다."
  (영어 표현이 한국어로 자연스럽게 바뀜)
```

### SCD가 어떻게 동작하는가

답변 생성 중 다음 단어를 선택할 때, 목표 언어(한국어)가 아닌 단어에 패널티를 준다:

```
[다음 단어 후보]
  "제안" (한국어) → 확률 0.35 → 페널티 없음 → 유지: 0.35
  "propose" (영어) → 확률 0.50 → beta=0.3 페널티 → 0.50 - 0.30 = 0.20
  "한" (한국어) → 확률 0.15 → 페널티 없음 → 유지: 0.15

결과: "제안"이 선택됨 (0.35 > 0.20 > 0.15)
```

beta 값이 클수록 영어 단어의 확률이 더 크게 줄어 한국어 일관성이 높아진다.

관련 코드: `backend/modules/scd_decoder.py`

## SSE 스트리밍

### 왜 필요한가

언어 모델이 긴 답변을 생성하는 데 20–30초가 걸릴 수 있다. SSE(Server-Sent Events) 없이 기다리면 화면이 그 시간 동안 멈춰 있는 것처럼 보인다.

SSE를 쓰면 단어가 생성되는 즉시 화면에 나타난다:

```
[0.0초] 답변 시작 — 화면에 아무것도 없음
[0.3초] "CAD"
[0.6초] "CAD는"
[0.9초] "CAD는 언어"
[1.2초] "CAD는 언어 모델이"
...
[8.0초] 답변 완료
```

사용자는 답변이 나오기 시작하는 순간부터 읽을 수 있다. 8초를 기다리지 않고 읽으면서 기다린다.

관련 코드: `backend/api/routers/chat.py` (`/api/chat/query/stream`)

## Judge API — LLM이 스스로 채점하기

### 왜 사람이 아닌 모델이 채점하는가

실험에서 수백–수천 개의 답변이 나온다. 한 명이 하루 100개를 채점한다고 해도, 1,000개를 채점하려면 10일이 필요하다. 언어 모델이 자동으로 채점하면 수십 분 안에 끝난다.

### 실제로 어떻게 채점하는가

```
[Judge 프롬프트]
  아래 컨텍스트와 답변을 보고 다음 레이블 중 하나만 출력하라:
  SUPPORTED / PARTIAL / UNSUPPORTED

  [컨텍스트]
  "alpha=0.5 achieved the best Faithfulness score of 87.3%..."

  [답변]
  "제공된 문맥에 따르면 이 예시에서는 alpha=0.5 설정이 가장 높은 Faithfulness를 보였다."

  Label:

[Judge 출력]
  SUPPORTED
```

레이블 의미:
- **SUPPORTED**: 답변이 컨텍스트와 일치함 → Faithfulness = 1.0점
- **PARTIAL**: 일부 맞고 일부 틀림 → Faithfulness = 0.5점
- **UNSUPPORTED**: 컨텍스트에 없는 내용 → Faithfulness = 0.0점

관련 코드: `backend/api/routers/chat.py`, `backend/evaluation/ragas_eval.py`
