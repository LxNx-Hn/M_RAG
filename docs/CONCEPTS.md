## 1. RAG (Retrieval-Augmented Generation)

**한줄 요약:** AI가 답변하기 전에 관련 자료를 먼저 찾아보는 기술

### 쉬운 비유: "오픈북 시험"

시험을 볼 때 두 가지 방식이 있습니다.

- **클로즈드북 시험** = 일반 LLM (ChatGPT 등)
  - 외운 것만으로 답을 씀
  - 기억이 부정확하면 틀린 답을 자신 있게 쓰기도 함 (할루시네이션)

- **오픈북 시험** = RAG
  - 교재를 펼쳐서 관련 부분을 찾고(Retrieval)
  - 그 내용을 바탕으로 답안을 작성(Generation)
  - 근거가 있으니 더 정확한 답이 나옴

RAG는 이 "오픈북 시험" 방식입니다. AI가 혼자 기억에만 의존하지 않고,
실제 문서에서 관련 정보를 검색(Retrieval)한 뒤 그것을 참고해서 답변을 생성(Generation)합니다.

### 왜 필요한가?

- LLM은 학습 데이터에 없는 최신 정보나 특정 논문 내용을 알 수 없음
- 기억만으로 답하면 "그럴듯하지만 틀린 답"(할루시네이션)이 나올 수 있음
- 근거 문서를 제시할 수 있어서 답변의 신뢰도가 높아짐

### M-RAG에서 어떻게 사용되나?

M-RAG의 이름 자체가 **Modular RAG**입니다. 사용자가 논문 PDF를 업로드하면:

1. 논문 내용을 잘게 쪼개서 저장해두고 (인덱싱)
2. 사용자 질문이 들어오면 관련 부분을 검색하고 (Retrieval)
3. 검색된 내용을 바탕으로 한국어 답변을 생성합니다 (Generation)

일반 RAG와 다른 점은 **"모듈러"** 라는 것입니다.
질문 유형에 따라 5개의 다른 파이프라인(A~E)으로 분기해서 처리합니다.

> **더 알고 싶다면:** "RAG survey paper", "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks (Lewis et al., 2020)"

## 2. 임베딩 (Embedding)

**한줄 요약:** 텍스트를 컴퓨터가 비교할 수 있는 숫자 벡터로 변환하는 기술

### 쉬운 비유: "텍스트를 지도 위 좌표로 변환"

서울, 부산, 대전, 도쿄가 있다고 생각해봅시다.

- 지도 위에서 서울과 대전은 가깝고, 서울과 도쿄는 멀죠?
- 이처럼 **의미가 비슷한 텍스트는 가까운 좌표에, 다른 텍스트는 먼 좌표에** 배치하는 것이 임베딩입니다.

예를 들어:
- "강아지가 공원에서 뛰어놀았다" -> [0.82, 0.15, 0.43, ...]
- "개가 잔디밭에서 놀고 있다" -> [0.80, 0.17, 0.41, ...] (비슷한 좌표!)
- "주가가 폭락했다" -> [0.12, 0.95, 0.03, ...] (전혀 다른 좌표)

실제로는 2차원 좌표가 아니라 1024차원의 숫자 배열(벡터)이지만,
원리는 지도와 같습니다.

### 왜 필요한가?

- 컴퓨터는 텍스트를 직접 이해하지 못하고, 숫자로 바꿔야 비교할 수 있음
- 단순 키워드 매칭이 아니라 **의미적 유사도**를 계산할 수 있음
  - "머신러닝 성능"을 검색하면 "기계학습 정확도"도 찾아줌
- 검색의 품질을 결정하는 핵심 요소

### M-RAG에서 어떻게 사용되나?

- 임베딩 모델: **BAAI/bge-m3** (한국어 + 영어 모두 잘 처리하는 다국어 모델)
- 논문 텍스트 청크를 1024차원 벡터로 변환
- 사용자 질문도 같은 모델로 벡터로 변환
- 두 벡터 간의 거리를 계산해서 가장 관련 있는 청크를 찾음
- `modules/embedder.py`에서 처리

> **더 알고 싶다면:** "word embedding", "sentence embedding", "BGE-M3 모델"

## 3. 벡터 DB (ChromaDB)

**한줄 요약:** 임베딩 벡터를 효율적으로 저장하고 빠르게 검색하는 전용 데이터베이스

### 쉬운 비유: "도서관 색인 카드 시스템"

도서관에서 책을 찾을 때를 생각해봅시다.

- 수만 권의 책이 있을 때 일일이 다 뒤지면 하루가 걸림
- 하지만 색인 카드 시스템이 있으면?
  - "이 주제는 3층 A구역 12번 서가에 있습니다"
  - 바로 원하는 책을 찾을 수 있음!

벡터 DB는 디지털 세계의 색인 카드 시스템입니다.

- 수천 개의 임베딩 벡터가 저장되어 있고
- "이 질문과 가장 가까운 벡터 20개 찾아줘"라고 하면
- 모든 벡터를 하나하나 비교하지 않고도 빠르게 찾아줌

### 왜 필요한가?

- 논문 하나만 해도 수백 개의 청크가 나옴. 여러 논문이면 수천~수만 개
- 일반 DB(MySQL 등)는 벡터 유사도 검색에 최적화되어 있지 않음
- 벡터 DB는 "가장 비슷한 것 찾기"에 특화된 인덱싱 구조를 사용

### M-RAG에서 어떻게 사용되나?

- **ChromaDB** 사용 (경량, 설치 간편, Python 네이티브)
- 논문 텍스트 청크의 임베딩 벡터를 `chroma_db/` 폴더에 저장
- 질문이 들어오면 상위 20개(TOP_K_RETRIEVAL=20) 유사 청크를 검색
- 각 청크에 메타데이터(논문명, 섹션, 페이지 번호 등)도 함께 저장
- `modules/vector_store.py`에서 처리

> **더 알고 싶다면:** "ChromaDB 공식 문서", "벡터 데이터베이스 비교 (Pinecone, Weaviate, Milvus)"

## 4. 청킹 (Chunking)

**한줄 요약:** 긴 문서를 LLM이 처리할 수 있는 작은 단위로 나누는 과정

### 쉬운 비유: "책을 메모 카드로 자르기"

시험 공부를 할 때, 교과서 전체를 통째로 가져가면 원하는 내용을 찾기 어렵습니다.
대신 핵심 내용을 메모 카드에 하나씩 적어두면:

- 각 카드는 하나의 주제에 집중 (적절한 크기)
- 빠르게 넘기면서 원하는 카드를 찾을 수 있음
- 너무 작으면 맥락이 없고, 너무 크면 불필요한 내용이 섞임

청킹도 마찬가지입니다.
논문 전체를 하나로 넣으면 LLM의 처리 한계를 넘기고,
한 문장씩 나누면 맥락이 사라집니다.
그래서 **적절한 크기로 나누되, 약간 겹치게** 합니다.

### 왜 필요한가?

- LLM은 한 번에 처리할 수 있는 텍스트 길이에 제한이 있음
- 검색 정확도를 높이려면 하나의 청크가 하나의 주제를 담아야 함
- 너무 크면 불필요한 내용 포함, 너무 작으면 문맥 손실

### M-RAG에서 어떻게 사용되나?

- 청크 크기: **512 토큰** (약 한국어 300자 정도)
- 겹침(오버랩): **64 토큰** (앞뒤 청크 사이에 내용이 약간 겹침)
- 최소 크기: **50 토큰** (너무 짧은 조각은 버림)
- 섹션 경계를 인식해서 다른 섹션 내용이 한 청크에 섞이지 않도록 함
- `modules/chunker.py`에서 처리

> **더 알고 싶다면:** "text chunking strategies", "RecursiveCharacterTextSplitter", "semantic chunking"

## 5. 리랭킹 (Reranking)

**한줄 요약:** 1차 검색 결과를 더 정밀한 모델로 재평가하여 순위를 조정하는 과정

### 쉬운 비유: "전문가가 검색 결과 재정렬"

인터넷 검색을 생각해봅시다.

1. **1차 검색 (Retrieval):** 구글이 수십억 페이지에서 관련 있어 보이는 1000개를 빠르게 뽑음
   - 빠르지만 대략적임
2. **리랭킹 (Reranking):** 해당 분야 전문가가 1000개 중 상위 결과를 꼼꼼히 읽고,
   "이건 진짜 관련 있고, 이건 겉보기만 비슷하네"라고 순위를 다시 매김
   - 느리지만 정확함

즉, **속도 vs 정확도의 2단계 전략**입니다.
1차에서 빠르게 후보를 줄이고, 2차에서 정밀하게 순위를 정합니다.

### 왜 필요한가?

- 임베딩 검색은 빠르지만, 미묘한 의미 차이를 놓칠 수 있음
- 리랭커는 질문과 문서를 함께 읽고 관련성을 판단 (Cross-Encoder 방식)
- 검색 정확도가 크게 향상됨 (상위 5개 결과의 품질이 중요!)

### M-RAG에서 어떻게 사용되나?

- 리랭킹 모델: **cross-encoder/ms-marco-MiniLM-L-6-v2**
- 1차 검색으로 20개 후보를 뽑고 (TOP_K_RETRIEVAL=20)
- 리랭커가 이 20개를 재평가해서 상위 5개만 선택 (TOP_K_RERANK=5)
- 하이브리드 검색(밀집 벡터 60% + BM25 키워드 40%)도 함께 사용
- `modules/reranker.py`에서 처리

> **더 알고 싶다면:** "cross-encoder reranking", "bi-encoder vs cross-encoder", "Reciprocal Rank Fusion (RRF)"

## 6. LLM (Large Language Model)

**한줄 요약:** 대규모 텍스트 데이터로 학습한, 자연어를 이해하고 생성할 수 있는 AI 모델

### 쉬운 비유: "엄청 많이 읽은 비서"

상상해봅시다. 비서가 있는데, 이 비서는:

- 인터넷에 있는 거의 모든 글을 읽었음 (수조 단어)
- 논문, 뉴스, 소설, 코드, 위키백과 등 모든 종류
- 이 엄청난 독서량을 바탕으로 질문에 답하고 글을 써줌

하지만 이 비서에게도 한계가 있습니다:
- 학습한 시점 이후의 정보는 모름 (지식 컷오프)
- 읽은 내용을 때때로 잘못 기억함 (할루시네이션)
- 특정 분야의 세부 내용은 부정확할 수 있음

그래서 RAG로 "참고 자료"를 제공해주면 훨씬 정확한 답을 만들 수 있습니다.

### 왜 필요한가?

- 검색만으로는 사용자에게 직접적인 답을 줄 수 없음
- 검색된 여러 조각을 종합하여 자연스러운 한국어 답변을 생성해야 함
- 질문의 의도를 이해하고, 적절한 형식으로 답변을 구성하는 능력이 필요

### M-RAG에서 어떻게 사용되나?

- 메인 생성 모델: **EXAONE-3.5-7.8B-Instruct** (LG AI 연구원이 개발한 한국어 특화 LLM)
- 베이스라인 모델: **Llama-3.1-8B-Instruct** (CAD에서 비교용으로 사용)
- 검색된 청크를 컨텍스트로 넣어주고, 질문에 대한 한국어 답변을 생성
- 온도(Temperature): 0.1 (창의성보다 정확성에 초점)
- 최대 생성 토큰: 1024

> **더 알고 싶다면:** "EXAONE 모델", "Transformer 아키텍처", "GPT vs BERT vs LLaMA 비교"

## 7. 프롬프트 엔지니어링

**한줄 요약:** AI에게 원하는 답을 얻기 위해 질문(프롬프트)을 잘 설계하는 기술

### 쉬운 비유: "올바른 질문 하기"

같은 사람에게 물어도, 질문 방식에 따라 답이 달라집니다.

- 나쁜 질문: "이 논문 설명해줘"
  - -> 너무 광범위해서 무엇을 원하는지 모름
- 좋은 질문: "이 논문의 핵심 방법론을 3가지로 요약하고, 각각 기존 방법과 어떻게 다른지 표로 정리해줘"
  - -> 구체적인 형식과 내용을 지정했으므로 원하는 답을 받을 수 있음

LLM에게도 마찬가지입니다.
프롬프트(지시문)를 어떻게 쓰느냐에 따라 답변의 품질이 크게 달라집니다.

### 왜 필요한가?

- 같은 LLM이라도 프롬프트에 따라 답변 품질 차이가 큼
- "너는 논문 리뷰 전문가야. 아래 맥락을 바탕으로 답해줘"처럼 역할을 부여하면 더 좋은 답이 나옴
- 출력 형식(한국어, 마크다운, 표 등)을 지정할 수 있음

### M-RAG에서 어떻게 사용되나?

- 5개 파이프라인(A~E)마다 목적에 맞는 프롬프트 템플릿이 다름
  - 파이프라인 A (단순 QA): "아래 맥락을 참고하여 질문에 답하세요"
  - 파이프라인 C (비교): "아래 두 논문의 공통점과 차이점을 분석하세요"
  - 파이프라인 E (요약): "아래 내용을 종합하여 전체 요약을 작성하세요"
- 검색된 청크(컨텍스트)를 프롬프트에 삽입하여 LLM에 전달
- 한국어 답변을 유도하는 지시문 포함

> **더 알고 싶다면:** "prompt engineering guide", "few-shot prompting", "chain-of-thought prompting"

## 8. CAD (Contrastive Decoding)

**한줄 요약:** 근거 없이 지어내는 경향을 억제하여 할루시네이션을 줄이는 디코딩 기법

### 쉬운 비유: "교과서로 팩트체크"

학생이 에세이를 쓸 때:

- **교과서를 안 보고 쓴 버전** (베이스라인 모델의 출력)
  - 기억에 의존 -> 사실과 다른 내용이 섞일 수 있음

- **교과서를 보면서 쓴 버전** (컨텍스트가 주어진 메인 모델의 출력)
  - 근거를 참고 -> 더 정확함

CAD는 이 두 버전을 비교합니다.
"교과서를 보고 쓴 내용" 중에서 "안 보고도 썼을 내용"을 약간 억제하고,
"교과서를 봐야만 나오는 내용"을 강화합니다.

결과적으로 **검색된 근거에 더 충실한 답변**이 만들어집니다.

### 왜 필요한가?

- LLM은 컨텍스트를 무시하고 자기가 아는 내용으로 답하는 경향이 있음
- 특히 학술 논문처럼 정확성이 중요한 경우, 근거 기반 답변이 필수
- CAD는 추가 학습 없이 디코딩 단계에서 할루시네이션을 줄일 수 있음

### M-RAG에서 어떻게 사용되나?

- 메인 모델(EXAONE)에 컨텍스트를 주고 확률 분포를 계산
- 베이스라인 모델(Llama)에 컨텍스트 없이 확률 분포를 계산
- 두 분포의 차이를 이용해서 최종 토큰을 선택
- 억제 강도: **CAD_ALPHA = 0.5** (0이면 효과 없음, 1이면 최대 억제)
- `modules/contrastive_decoder.py`에서 처리

> **더 알고 싶다면:** "Context-Aware Decoding (CAD)", "contrastive decoding for hallucination reduction"

## 9. HyDE (Hypothetical Document Embeddings)

**한줄 요약:** 질문에 대한 가상의 답변을 먼저 만들고, 그것으로 검색하는 기법

### 쉬운 비유: "답을 상상해서 찾기"

도서관에서 자료를 찾을 때:

- **일반 검색:** "Transformer의 어텐션 메커니즘은?"이라는 질문으로 검색
  - 질문 형태의 텍스트와 답변 형태의 문서는 형태가 다름
  - 매칭이 잘 안 될 수 있음

- **HyDE 방식:** LLM에게 먼저 "대략적인 답변"을 상상하게 함
  - "Transformer의 어텐션 메커니즘은 Query, Key, Value를 이용하여..."
  - 이 가상 답변으로 검색하면 실제 논문 내용과 형태가 비슷!
  - 더 정확한 문서를 찾을 수 있음

즉, **질문을 답변처럼 바꿔서 검색 품질을 높이는 트릭**입니다.

### 왜 필요한가?

- 질문과 문서는 문체가 다름 (의문문 vs 서술문)
- 질문을 그대로 검색하면 의미적으로 가장 관련 있는 문서를 놓칠 수 있음
- 가상 답변은 실제 문서와 더 비슷한 형태이므로 임베딩 공간에서 더 가까움

### M-RAG에서 어떻게 사용되나?

- Query Expander 모듈에서 HyDE 기법을 활용
- 사용자 질문 -> LLM이 가상 답변 생성 -> 가상 답변으로 벡터 검색
- 원래 질문으로 한 검색 결과와 합쳐서 더 다양한 관련 문서를 확보
- `modules/query_expander.py`에서 처리

> **더 알고 싶다면:** "HyDE paper (Gao et al., 2022)", "query expansion techniques"

## 10. vLLM / GPU 서빙

**한줄 요약:** LLM을 GPU에서 빠르게 실행하기 위한 고성능 추론 엔진

### 쉬운 비유: "고속 조립 라인"

자동차 공장을 상상해봅시다.

- **수작업 조립** = 기본 Python 추론
  - 한 명이 차를 처음부터 끝까지 만듦
  - 느리지만 작동은 함

- **고속 조립 라인** = vLLM
  - 컨베이어 벨트 위에서 여러 공정이 동시에 진행
  - 부품을 미리 준비해두고 (KV 캐시)
  - 여러 대를 동시에 조립 (배치 처리)
  - 같은 시간에 훨씬 많은 차를 생산

vLLM도 마찬가지입니다. 같은 GPU에서 LLM을 2~5배 더 빠르게 실행합니다.

### 왜 필요한가?

- 7.8B 파라미터 모델은 일반적으로 실행이 느림
- 사용자가 질문할 때마다 수십 초씩 기다리면 서비스 품질이 떨어짐
- GPU 메모리를 효율적으로 사용해야 여러 요청을 동시에 처리 가능
- PagedAttention 같은 기술로 메모리 낭비를 줄여줌

### M-RAG에서 어떻게 사용되나?

- EXAONE 모델과 Llama 모델을 GPU에서 서빙
- Docker 컨테이너에서 NVIDIA GPU 1개를 할당받아 실행
- 연속 배치(continuous batching)로 여러 파이프라인 요청을 효율적으로 처리
- docker-compose.yml에서 GPU 리소스를 예약

> **더 알고 싶다면:** "vLLM 공식 문서", "PagedAttention 논문", "LLM inference optimization"

## 11. FastAPI

**한줄 요약:** Python으로 빠르게 웹 API를 만들 수 있는 프레임워크

### 쉬운 비유: "레스토랑 웨이터"

레스토랑에서:

- **손님** = 프론트엔드 (사용자 화면)
- **웨이터** = FastAPI (요청을 받고 전달)
- **주방** = 백엔드 로직 (RAG 파이프라인, LLM 등)

웨이터의 역할:
1. 손님의 주문을 받음 (API 요청 수신)
2. 주문이 올바른지 확인 (입력 검증)
3. 주방에 주문을 전달 (파이프라인 호출)
4. 요리가 나오면 손님에게 서빙 (응답 반환)

웨이터가 없으면 손님이 직접 주방에 들어가야 하는데,
그러면 주방이 엉망이 되겠죠? API도 마찬가지입니다.

### 왜 필요한가?

- 프론트엔드(화면)와 백엔드(로직)를 분리하면 각각 독립적으로 개발/수정 가능
- 여러 클라이언트(웹, 모바일, 다른 서비스)가 같은 API를 사용할 수 있음
- 자동으로 API 문서(Swagger UI)를 생성해줌
- 비동기(async) 처리로 동시에 여러 요청을 효율적으로 처리

### M-RAG에서 어떻게 사용되나?

- 현재 M-RAG는 Streamlit으로 UI와 로직이 통합되어 있음 (`app.py`)
- FastAPI는 향후 프론트엔드-백엔드 분리 시 사용 가능
- 예: React/Next.js 프론트엔드에서 FastAPI 백엔드의 `/api/query` 엔드포인트를 호출
- 현재 프론트엔드 폴더(`frontend/`)가 이미 존재하며, API 기반 아키텍처로 확장 가능

> **더 알고 싶다면:** "FastAPI 공식 문서", "REST API란?", "Streamlit vs FastAPI 비교"

## 12. Docker

**한줄 요약:** 애플리케이션을 어디서든 동일하게 실행할 수 있도록 패키징하는 기술

### 쉬운 비유: "소프트웨어 배송 컨테이너"

해외에서 물건을 주문하면 배송 컨테이너에 담겨서 옵니다.

- **컨테이너가 없다면:**
  - "이 물건은 온도 22도에서 보관해야 해요"
  - "이 물건은 습도가 낮아야 해요"
  - "이 물건 옆에 저 물건을 두면 안 돼요"
  - 배송이 매우 복잡해짐

- **컨테이너가 있다면:**
  - 필요한 모든 조건을 컨테이너 안에 미리 세팅
  - 어떤 배에 싣든, 어떤 항구에 도착하든 내부 조건은 동일
  - "이 컨테이너만 가져다 놓으면 됩니다"

Docker도 마찬가지입니다. M-RAG를 실행하려면 Python, CUDA, ChromaDB, 각종 라이브러리...
이 모든 것을 Docker 컨테이너 안에 넣어두면, 어떤 컴퓨터에서든 `docker compose up` 한 줄로 실행됩니다.

### 왜 필요한가?

- "제 컴퓨터에서는 되는데 다른 컴퓨터에서 안 돼요" 문제를 해결
- Python 버전, 라이브러리 버전 등 환경을 완벽히 동일하게 유지
- 팀원, 교수님, 심사위원 누구나 쉽게 프로젝트를 실행할 수 있음
- GPU 환경도 컨테이너 내에서 관리 가능 (nvidia-docker)

### M-RAG에서 어떻게 사용되나?

- `docker-compose.yml`로 전체 시스템을 정의
- 포트 8501로 Streamlit UI를 외부에 노출
- GPU 1개를 컨테이너에 할당 (NVIDIA 드라이버 연동)
- `data/`와 `chroma_db/` 볼륨을 마운트하여 데이터 영속성 보장
- HuggingFace 모델 캐시를 볼륨으로 관리하여 재다운로드 방지

> **더 알고 싶다면:** "Docker 공식 문서", "Docker Compose 튜토리얼", "nvidia-docker GPU 설정"

## 13. 시스템 아키텍처

**한줄 요약:** M-RAG의 13개 모듈이 5개 파이프라인으로 조합되어 질문에 답하는 전체 구조

### 전체 흐름도 (ASCII Diagram)

```
사용자 질문
    |
    v
+---------------------------+
|   Query Router (모듈 6)    |   질문을 분석해서 최적의 파이프라인을 선택
+---------------------------+
    |
    +----+----+----+----+
    |    |    |    |    |
    v    v    v    v    v
   [A]  [B]  [C]  [D]  [E]    <-- 5개 파이프라인
    |    |    |    |    |
    v    v    v    v    v
+---------------------------+
|   공통 모듈들               |
|                           |
|  PDF Parser (모듈 1)       |   PDF에서 텍스트 추출
|  Section Detector (모듈 2) |   논문 섹션 구분 (초록, 방법론, 결과...)
|  Chunker (모듈 3)          |   텍스트를 512 토큰 청크로 분할
|  Embedder (모듈 4)         |   텍스트를 1024차원 벡터로 변환
|  Vector Store (모듈 5)     |   ChromaDB에 벡터 저장/검색
+---------------------------+
    |
    v
+---------------------------+
|   검색 & 정제               |
|                           |
|  Query Expander (모듈 7)   |   HyDE로 질문 확장
|  Hybrid Retriever (모듈 8) |   벡터 + BM25 하이브리드 검색
|  Reranker (모듈 9)         |   Cross-Encoder로 상위 5개 선별
|  Context Compressor (모듈10)|   컨텍스트 압축 (3072 토큰 이내)
+---------------------------+
    |
    v
+---------------------------+
|   생성 & 검증               |
|                           |
|  LLM Generator            |   EXAONE-3.5로 한국어 답변 생성
|  Contrastive Decoder (11) |   CAD로 할루시네이션 억제
|  Citation Tracker (모듈12) |   인용 정보 추적 (arXiv 연동)
|  RAGAS Evaluator (모듈13)  |   답변 품질 자동 평가
+---------------------------+
    |
    v
사용자에게 답변 + 근거 표시
```

### 5개 파이프라인 상세

각 파이프라인은 질문 유형에 맞게 모듈을 다르게 조합합니다.

```
파이프라인 A: 단순 QA
  질문 -> 벡터 검색 -> 리랭킹 -> LLM 생성 -> 답변
  예: "이 논문에서 사용한 데이터셋이 뭔가요?"

파이프라인 B: 섹션 특화
  질문 -> 섹션 키워드 감지 -> 해당 섹션만 필터 검색 -> 리랭킹 -> LLM 생성 -> 답변
  예: "결과 섹션에서 F1 스코어가 어떻게 되나요?"

파이프라인 C: 멀티 논문 비교
  질문 -> 논문별 병렬 검색 -> 결과 합성 -> LLM 비교 분석 생성 -> 답변
  예: "BERT와 GPT 논문의 방법론 차이를 비교해줘"

파이프라인 D: 인용 트래커
  질문 -> arXiv/Semantic Scholar에서 참고문헌 수집 -> 확장 검색 -> 답변
  예: "이 논문이 인용한 주요 논문들은?"

파이프라인 E: 전체 요약
  질문 -> 계층적 검색 (RAPTOR 스타일) -> 전체 요약 생성 -> 답변
  예: "이 논문 전체를 요약해줘"
```

### 동작 흐름 워크스루 (예시)

사용자가 "이 논문의 실험 결과에서 기존 방법보다 얼마나 좋아졌나요?"라고 질문하면:

**1단계: 라우팅**
- Query Router가 "결과", "성능" 키워드를 감지
- 파이프라인 B (섹션 특화)로 분기

**2단계: 검색**
- "result" 섹션의 청크만 필터링하여 검색 범위를 좁힘
- 하이브리드 검색: 벡터 유사도(60%) + BM25 키워드 매칭(40%)으로 20개 후보 검색
- 리랭커가 20개를 재평가하여 상위 5개 선별

**3단계: 컨텍스트 준비**
- Context Compressor가 5개 청크를 3072 토큰 이내로 정리
- 프롬프트 템플릿에 컨텍스트를 삽입

**4단계: 답변 생성**
- EXAONE 모델이 컨텍스트를 참고하여 한국어 답변 생성
- CAD가 Llama 모델과 비교하여 할루시네이션을 억제
- 최종 답변에 근거 청크의 출처 표시

**5단계: 결과 반환**
- Streamlit UI에 답변과 함께 참고한 논문 부분을 표시
- 사용자가 근거를 직접 확인할 수 있음

### 기술 스택 요약

```
[프론트엔드]   Streamlit (+ React 프론트엔드 개발 중)
[백엔드]       Python 3.x
[임베딩]       BAAI/bge-m3 (1024차원)
[벡터 DB]      ChromaDB
[LLM]          EXAONE-3.5-7.8B-Instruct
[베이스라인]    Llama-3.1-8B-Instruct
[리랭커]       ms-marco-MiniLM-L-6-v2
[서빙]         vLLM + NVIDIA GPU
[컨테이너]     Docker + Docker Compose
[평가]         RAGAS (자동 평가 프레임워크)
```

M-RAG는 이 12가지 기술을 모듈 단위로 조합한 시스템입니다.
각 모듈은 독립적으로 교체하거나 개선할 수 있어서 "모듈러(Modular)"라는 이름이 붙었습니다.

처음에는 전체 그림이 복잡해 보이지만, 핵심 흐름은 간단합니다:

> **논문 업로드 -> 잘게 쪼개서 저장 -> 질문이 오면 관련 부분 검색 -> 검색 결과로 답변 생성**

나머지 모듈들(리랭킹, CAD, HyDE 등)은 이 기본 흐름의 **각 단계를 더 정확하게** 만들어주는 부품들입니다.

궁금한 점이 있다면 각 모듈의 소스 코드(`modules/` 폴더)를 직접 살펴보거나,
위에서 추천한 키워드로 검색해보세요!
