# M-RAG 쉬운 설명서

> "이 시스템이 정확히 뭐 하는 거야?"라고 물을 때 바로 펼쳐보는 문서
> 문서 기준 2026-04-26, 비전공자와 새 팀원의 빠른 온보딩 목적

---

## 이 시스템은 뭐 하는 도구야?

논문을 올리면, 그 논문을 읽고 근거를 찾아 답하고 환각을 줄이도록 설계된 AI 도우미입니다.
그냥 대화만 하는 챗봇이 아니라, 논문 안에서 근거를 찾고 답하도록 설계되어 있습니다.

---

## 일반 챗봇이랑 뭐가 달라?

비유하면 오픈북 시험에 가깝습니다.

- 일반 챗봇: 학습된 기억으로 먼저 답하려는 성향이 큼
- M-RAG: 문서에서 관련 부분을 먼저 찾고, 그다음 답을 생성
- 결과적으로 답변에 출처와 근거를 붙이기 쉬움

---

## 왜 모듈형(모듈러)인가?

질문이 모두 같은 유형이 아니기 때문입니다.
병원 접수처가 증상에 따라 과를 나누듯, 질문 의도에 맞는 경로를 다르게 탑니다.

- 결과 섹션만 묻는 질문
- 두 논문을 비교하는 질문
- 참고문헌을 따라가는 질문
- 전체 요약을 만드는 질문
- 퀴즈를 만드는 질문

---

## 내부에서는 어떤 순서로 움직여?

아래 흐름으로 동작합니다.

1. 문서를 작은 조각으로 나눔
2. 각 조각을 검색 가능한 형태로 저장
3. 질문을 보고 적절한 경로를 선택
4. 관련 조각을 찾고 정렬
5. 필요한 문맥만 압축
6. LLM이 최종 답변 생성

---

## Mini와 Base는 어떻게 골라?

- Mini: 로컬 12GB급 GPU에 맞춘 기본 모델
- Base: 더 큰 GPU에서 비교 실험이나 대형 추론에 쓰는 선택 모델
- 둘 다 유지하지만 기본 실행 경로는 Mini

---

## 왜 양자화를 쓰지 않나?

- 현재 로컬 기준과 문서 기준을 단순하게 유지하기 위해
- Mini만으로도 연구용 검증 경로를 운영할 수 있기 때문

---

## 그럼 외부 LLM 서버나 vLLM은 왜 바로 안 쓰나?

- 지금 논문 경로의 핵심은 단순 생성이 아니라 CAD와 SCD를 포함한 생성 제어이기 때문
- 그래서 이번 구현에서는 외부 상용 LLM API 연동이나 `vLLM` 전환을 바로 진행하지 않음
- 이유는 현재 클레임이 환각 억제와 언어 이탈 억제를 포함한 생성 제어 효과에 있기 때문
- OpenAI 같은 외부 API는 연결은 쉽지만 생성 중간 제어를 그대로 옮기기 어려움
- `vLLM` 도 일반 생성 서빙만 보면 좋은 선택지지만 CAD와 SCD까지 유지하려면 추가 구현이 필요함
- 그래서 현재 기준 연구 경로에서는 내부 생성 구조를 유지하고, 서비스 경로에서만 별도 선택지로 검토하는 편이 안전함
- `vLLM` 기반 RAG에서 환각 억제 기법을 유지하는 방향은 이번 구현이 아니라 다음 단계 연구 주제로 분리하는 편이 적절함
