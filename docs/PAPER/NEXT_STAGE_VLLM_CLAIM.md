# M-RAG 다음 단계 연구 클레임 후보 vLLM 경로

- 기준일 2026-04-26
- 목적 현재 논문 이후 확장 가능한 후속 연구 클레임을 현재 코드 기준으로 정리
- 원칙 현재 논문의 핵심 주장과 혼합하지 않음

## 이 문서가 필요한 이유

- 현재 논문 클레임은 한국어 중심 학술 문서 QA에서 CAD와 SCD를 포함한 생성 제어의 효과를 설명하는 데 초점
- `vLLM` 경로는 서빙 효율과 대규모 추론 구조를 다루는 별도 문제
- 따라서 `vLLM` 경로는 현재 논문의 직접 결과가 아니라 다음 단계 연구 후보로 분리하는 편이 안전

## 현재 논문 클레임과 다음 단계 클레임의 차이

### 현재 논문 클레임

- 라우터 기반 모듈형 학술 문서 QA 구조
- 하이브리드 검색과 재정렬 스택
- CAD와 SCD 기반 생성 제어
- 로컬 실행 기준 재현 가능한 실험 경로

### 다음 단계 클레임 후보

- `vLLM` 기반 고효율 추론 환경에서도 환각 억제와 언어 이탈 억제를 유지할 수 있는가
- 또는 `vLLM` 의 배치 서빙 구조에 맞는 새로운 생성 제어 기법을 설계할 수 있는가

## 핵심 문제 정의

- 외부 상용 LLM API는 연결은 쉽지만 생성 내부 단계 제어가 어려움
- 현재 CAD와 SCD는 `transformers` 기반 `LogitsProcessor` 경로에 직접 결합
- `vLLM` 은 일반 생성 서빙에는 유리하지만 현재 CAD와 SCD를 그대로 옮기기에는 추가 설계가 필요
- 따라서 다음 단계 연구는 단순 서빙 엔진 교체가 아니라 서빙 효율과 생성 제어를 동시에 만족시키는 구조를 다루게 됨

## 추천 클레임 문장 후보

### 후보 1

- `vLLM` 기반 RAG 서빙 환경에서 한국어 중심 학술 QA를 위한 환각 억제형 생성 제어를 재구성하고 그 효과를 검증

### 후보 2

- 배치 기반 고효율 추론 엔진에서 CAD와 SCD 계열 생성 제어를 유지 또는 재설계하여 서빙 효율과 응답 품질을 함께 개선

### 후보 3

- 일반 생성 서빙 최적화와 환각 억제형 디코딩 제어 사이의 구조적 충돌을 분석하고 `vLLM` 에 적합한 제어 경로를 제안

## 왜 새로운 클레임으로 볼 수 있는가

- 현재 클레임은 품질 중심 연구
- 다음 단계 클레임은 품질과 서빙 효율의 동시 최적화 연구
- 현재 구현은 단일 프로세스 또는 직접 로딩 구조에 가까움
- `vLLM` 경로는 배치 처리, 동시성, 서빙 추상화, 외부 추론 분리를 전제로 함
- 따라서 같은 CAD와 SCD 이름을 유지하더라도 연구 질문 자체가 달라짐

## 기술적으로 달라지는 지점

### 현재 구조

- `backend/modules/generator.py` 가 모델과 토크나이저를 직접 로드
- `backend/modules/cad_decoder.py` 가 문서 없는 프롬프트 logits와 KV cache를 함께 사용
- `backend/modules/scd_decoder.py` 가 한국어 외 토큰에 직접 패널티 적용
- A부터 F까지 파이프라인이 이 내부 생성 제어를 직접 호출

### vLLM 경로에서 달라지는 점

- 생성 엔진이 앱 서버 밖으로 분리될 가능성이 큼
- 요청은 OpenAI 호환 API 또는 `vLLM` 전용 경로를 통해 전달
- 배치 단위 처리와 동시성 최적화가 기본 전제
- 제어 로직은 `vLLM` 의 custom logits processor 또는 별도 제어 계층으로 다시 맞춰야 함

## 난이도 구분

### 낮음

- plain generation 기준 `vLLM` 서빙 연결
- Mini와 Base 모델 교체
- 스트리밍 응답 연결
- RunPod 또는 원격 GPU 운영 자동화

### 중간

- `SCD` 유사 제어를 `vLLM` 경로에 이식
- 한국어 이탈률과 응답 속도 사이의 균형 재조정

### 높음

- `CAD` 유사 제어를 `vLLM` 경로에 맞게 재현
- 문서 없는 프롬프트 분기와 배치 처리 구조를 함께 다루는 방식 설계
- 서빙 효율 저하 없이 제어 강도를 유지하는 구현 검증

## 단계별 연구 전략

### 단계 1 서빙 분리 검증

- plain generation 기준 `vLLM` 경로 연결
- 현재 RAG 앞단 검색 파이프라인은 그대로 유지
- 응답 속도, 처리량, 동시성, 메모리 사용량 비교

### 단계 2 SCD 이식 검증

- 한국어 중심 응답 유지 목적의 토큰 제어를 `vLLM` 경로에 부분 이식
- 언어 이탈률과 응답 품질 변화 비교

### 단계 3 CAD 재설계 검증

- 문서 없는 프롬프트 대비 제어를 `vLLM` 배치 구조에 맞게 다시 정의
- 기존 CAD와 동일 효과를 목표로 하기보다 기능적으로 동등한 억제 효과를 검증

### 단계 4 통합 클레임 정리

- `vLLM` plain generation 대비
- `vLLM + SCD`
- `vLLM + CAD/SCD`
- 현재 `transformers + CAD/SCD`
- 위 네 축을 함께 비교

## 검증 지표 후보

### 품질 지표

- Faithfulness
- Answer Relevancy
- Context Precision
- Context Recall
- 수치 환각률
- 한국어 이탈률

### 시스템 지표

- 평균 응답 시간
- 첫 토큰 지연 시간
- 초당 토큰 처리량
- 동시 요청 수 기준 처리량
- GPU 메모리 사용량
- 요청 실패율

## 실험 비교군 추천

- 현재 경로 `transformers + CAD/SCD`
- `vLLM` plain generation
- `vLLM + SCD`
- `vLLM + CAD 유사 제어`
- 외부 상용 API plain generation

## 성공 시 주장 가능한 가치

- 서빙 효율과 생성 제어의 동시 달성 가능성 제시
- 한국어 중심 학술 QA에서 배치 서빙 구조와 제어형 디코딩의 양립 가능성 제시
- 단순히 더 빠른 서빙이 아니라 제어 가능한 고효율 RAG 경로 제시

## 실패해도 남는 연구 가치

- 어떤 제어는 외부 API에서 유지되기 어려운지 설명 가능
- `CAD` 와 `SCD` 중 무엇이 서빙 엔진 이동에 더 민감한지 정리 가능
- 서빙 효율과 생성 제어 사이의 trade-off 자체를 연구 결과로 남길 수 있음

## 지금 당장 주장하면 안 되는 것

- `vLLM` 기반 환각 억제가 이미 구현되었다는 표현
- 현재 저장소가 `vLLM` 을 지원한다는 표현
- `vLLM` 경로에서 CAD와 SCD 효과가 이미 검증되었다는 표현
- 외부 상용 API보다 항상 우수하다는 표현

## 현재 저장소와 연결되는 출발점
