# 경쟁 분석: M-RAG vs 유사 시스템

> 이 문서는 M-RAG의 학술 기여(C1~C4)를 경쟁 시스템과 비교하여 포지셔닝을 명확히 합니다.  
> 졸업논문 심사 준비 및 향후 논문 작성 시 참조용.

---

## 1. 비교 대상

| 시스템 | 출처 | 특징 |
|--------|------|------|
| **M-RAG** (본 시스템) | 본 졸업작품 | 한국어 학술 논문 특화 Modular RAG |
| **NotebookLM** | Google (2023~) | 범용 문서 QA, 상용 클로즈드 시스템 |
| **SciRAG** | arXiv 2511.14362 (2025) | 학술 문서 특화 RAG, 인용 인식 |
| **PaperQA2** | FutureHouse (2024) | 오픈소스 학술 RAG |
| **CG-RAG** | SIGIR 2024 | 인용 그래프 기반 RAG |

## 3. NotebookLM 심층 분석

### 3.1 공개된 기술 정보

NotebookLM이 공개한 기술 스택은 **웹 전달 레이어**에 한정됩니다:

```
공개된 기술 (오픈소스 고지 기준):
  프론트엔드: Angular, TypeScript, RxJS, Material Components, D3.js
  빌드 도구: Autoprefixer, CSSnano, SVGO, Sass
  UI 라이브러리: Lottie (애니메이션), Quill (텍스터 에디터), KaTeX (수식 렌더링), ProseMirror
  인프라: TensorFlow Serving, Firebase/Firestore

미공개: RAG 파이프라인 구조, 청킹 전략, 검색 알고리즘,
        라우팅 로직, 환각 억제 방법, 임베딩 모델
```

즉, 어떤 RAG 기법을 쓰는지, 라우팅이 있는지조차 공개되지 않았습니다.

### 3.2 학술 용도에서의 구조적 한계 (7가지)

1. **arXiv/학술 DB 연동 없음**: 인용 논문 자동 수집·인덱싱 불가. 사용자가 직접 업로드해야 함.
2. **섹션 인식 없음**: 논문의 Abstract/Method/Result/Conclusion을 구분하지 않아 정밀 섹션 검색 불가.
3. **인용 그래프 탐색 없음**: "이 논문이 인용한 논문들의 내용은?" 같은 쿼리에서 코퍼스 자동 확장 불가.
4. **수식/표 이해 취약**: KaTeX로 렌더링은 가능하나, 수식의 의미 이해나 수식 기반 QA 불가.
5. **학술 인용 형식 내보내기 없음**: BibTeX, RIS 등 논문 작성용 인용 형식 생성 불가.
6. **라우팅 불투명**: 어떤 전략으로 답변을 생성했는지 사용자가 알 수 없음. 연구자가 결과를 신뢰/재현하기 어려움.
7. **환각 억제 제어 불가**: CAD의 α 값처럼 사용자가 환각 억제 강도를 조절하는 인터페이스 없음.

### 3.3 "범용 문서 지원"이 M-RAG를 무력화하지 않는 이유

NotebookLM이 일반 문서(Word, 슬라이드, URL)를 지원하더라도, 이것이 학술 논문 특화 기능을 대체할 수 없습니다:

**논문 도메인에서만 유효한 M-RAG 기능:**

| 논문 특성 | M-RAG 대응 기능 | 일반 문서에서? |
|-----------|----------------|---------------|
| Reference 섹션 존재 | Pipeline D: 인용 자동 추적·인덱싱 | 없음 |
| 섹션 구조 (IMRaD) | Pipeline B: 섹션 필터 검색 | 없음 |
| 영문 논문 + 한국어 사용자 갭 | BGE-M3 크로스링구얼 + SCD | 해당 없음 |
| 수치 실험 결과 | CAD: 파라메트릭 지식 억제 | 효과 감소 |

M-RAG는 일반 문서도 처리합니다 (섹션 인식 실패 시 Fixed 청킹 + Pipeline A fallback). 이것은 설계된 동작이며, 논문 특화 기능이 비활성화될 뿐 시스템은 정상 동작합니다.

## 3.4 강의/교재 PDF 처리 (NotebookLM 동등 + α)

NotebookLM이 강의 PDF(BNF 문법, EBNF, 코드 블록, 다중 컬럼)에서 구조 보존 추출을 수행하는 것이 확인됨. M-RAG는 `pymupdf4llm` 기반 구조 보존 추출로 동등 기능을 확보.

| 기능 | M-RAG | NotebookLM |
|------|-------|-----------|
| 코드 블록 보존 (BNF/EBNF) | pymupdf4llm markdown 펜스 → `block_type="code"` | 있음 |
| 다중 컬럼 처리 | pymupdf4llm 자동 병합 | 있음 |
| 표 보존 | pymupdf4llm markdown 표 + pymupdf `find_tables()` fallback | 있음 |
| 수식 보존 | 폰트/유니코드 기반 `block_type="math"` | 있음 (KaTeX 렌더링) |
| **구조 블록 분할 금지** | chunker에서 code/table/math 분할 금지 → LLM이 BNF 문법을 깨뜨리지 않고 인용 | **없음 (불명)** |
| 정의/정리/증명 섹션 감지 | `LECTURE_PATTERNS` (definition/theorem/proof) | 없음 |

**M-RAG 추가 우위**: chunker 단에서 구조 블록 분할 금지 — BNF 문법, 알고리즘 의사코드, 수학 정리가 청크 경계에서 잘리지 않음.

## 3.5 특허 문서 처리 (M-RAG 고유 기능)

NotebookLM은 일반 문서로서 특허 PDF를 처리할 수 있지만, 특허 도메인 특화 기능은 없음.

| 기능 | M-RAG | NotebookLM |
|------|-------|-----------|
| 특허 섹션 자동 인식 (청구항/배경기술/상세한 설명) | `PATENT_PATTERNS` 11종 | 없음 |
| 인용 특허 자동 파싱 | `patent_tracker.parse_cited_patents()` | 없음 |
| Google Patents 메타데이터 수집 | `fetch_from_google_patents()` | 없음 |
| KIPRIS API 연동 (한국 특허) | `fetch_from_kipris()` (API 키 설정 시) | 없음 |
| 유사 특허 검색 | 청구항 키워드 기반 `search_similar_patents()` | 없음 |

## 4. SciRAG (arXiv 2511.14362) 포지셔닝

SciRAG는 가장 유사한 공개 경쟁 시스템입니다. M-RAG와 차이:

| 측면 | M-RAG | SciRAG |
|------|-------|--------|
| 환각 억제 | CAD (decoding-time) | 없음 |
| 언어 지원 | 한국어 특화 (SCD) | 영어 중심 |
| 라우팅 | 5-way 동적 분기 (키워드 스코어) | 단일 파이프라인 |
| 인용 추적 | arXiv 자동 다운로드+인덱싱 | 인용 인식 합성 (다운로드 없음) |
| 오픈소스 | 전체 코드 공개 | 논문만 공개 |

---

## 5. CG-RAG (SIGIR 2024) 포지셔닝

CG-RAG는 인용 그래프 다중 홉 추론이 강점이나:

- **영어 전용**, 한국어 미지원
- 환각 억제 메커니즘 없음
- 그래프 구축을 오프라인 전처리로 수행 (M-RAG는 쿼리 시 on-demand 확장)
- 단일 파이프라인 (라우팅 없음)

---

## 6. M-RAG의 방어 가능한 경쟁 위치

### 핵심 주장

> M-RAG는 **한국어 학술 논문**이라는 좁고 명확한 도메인에 특화되어, 어떤 경쟁 시스템도 동시에 갖추지 못한 다음 조합을 구현합니다:
>
> **섹션 인식 청킹 + 5-파이프라인 동적 라우팅 + CAD 환각 억제 + SCD 언어이탈 방지 + arXiv 자동 인덱싱**
>
> 특히 CAD를 한국어 학술 RAG에 적용·평가한 선행 연구가 없어, C3 기여는 독자적인 실험적 기여입니다.

### 투명성 차별화

M-RAG는 쿼리당 실행된 파이프라인 경로(A~E)와 각 처리 단계를 UI에 표시합니다. 연구자는:
- 왜 이 답변이 생성되었는지 추적 가능
- CAD α 값을 직접 조절하여 환각 억제 강도 제어 가능
- 소스 코드를 통해 검색 전략 전체를 감사(audit) 가능

이는 블랙박스 상용 시스템인 NotebookLM과 근본적으로 다른 연구 도구로서의 포지션을 의미합니다.
