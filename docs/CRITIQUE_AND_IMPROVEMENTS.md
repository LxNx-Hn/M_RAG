# M-RAG: 현 시스템 한계 분석 및 개선 로드맵

> 이 문서는 M-RAG의 현재 한계를 솔직하게 분석하고, 단계별 개선 방향을 제시합니다.  
> 졸업논문 "Limitations and Future Work" 섹션의 기초 자료로 사용합니다.

---

## 0. arXiv 미등록 논문 및 출판본-프리프린트 불일치 문제

**실제 상황**: Pipeline D(인용 추적)는 arXiv API를 사용하므로, arXiv에 등록된 논문만 자동 수집이 가능합니다.

**문제의 규모**:
- CS/ML/AI 분야: arXiv 등록률 ~70~80%. 그나마 나은 편.
- 생명과학/의학: ~30~50%. Nature, Cell 등 대형 저널 논문 상당수 비공개.
- 사회과학/인문학: ~10~20%. arXiv 이용률 자체가 낮음.
- 학회 워크샵 페이퍼, 산업 보고서: 거의 없음.

**프리프린트 vs 출판본 불일치**:
- arXiv 프리프린트와 피어리뷰 후 게재 최종본은 수치·결론이 다를 수 있음.
- 예: 초기 arXiv 버전에서 "95.2% accuracy"가 최종본에서 "94.8%"로 수정된 사례 존재.
- M-RAG는 이 불일치를 감지하거나 경고하는 기능이 없음.

**현실적 대응 방안** (과장 없이):

| 상황 | 현재 M-RAG | 구현 가능한 개선 |
|------|-----------|----------------|
| arXiv에 없는 논문 | 조용히 실패, 결과 없음 | "arXiv 미등록 — 직접 업로드 필요" 메시지 표시 |
| DOI만 있는 논문 | 접근 불가 | Semantic Scholar API로 메타데이터만 가져오기 (PDF X) |
| 프리프린트/출판본 불일치 | 감지 불가 | 해결 불가 수준 — 사용자에게 버전 확인 권고 메시지 |
| IEEE/ACM/Springer 유료 논문 | 접근 불가 | 사용자가 PDF 직접 업로드하면 처리 가능 |

**핵심 현실**: arXiv 미등록 논문은 자동 수집이 구조적으로 불가능합니다. Semantic Scholar API를 추가하면 메타데이터(제목, 저자, 초록, 인용 수)는 가져올 수 있지만, 전문(full text) PDF는 유료 논문의 경우 합법적으로 자동 취득이 어렵습니다. 이것은 M-RAG만의 문제가 아니라 모든 오픈소스 학술 RAG 시스템이 공유하는 구조적 한계입니다.

**단, 강의/교재/특허/기술문서 모드에서는 arXiv 의존도가 0이므로 이 한계가 비활성화됩니다.** Pipeline D(인용 추적)는 학술 논문 모드에서는 arXiv, 특허 모드에서는 Google Patents/KIPRIS로 분기되며, 강의/일반 문서 모드에서는 인용 추적이 라우팅되지 않습니다.

## 1. 현 시스템 한계 (L1~L8)

### L1: 수식 및 방정식 처리 — 해결됨 (텍스트 레벨)

pymupdf4llm + 폰트/유니코드 기반 수식 감지로 해결. Math 폰트(CMSY/STIX/Symbol) 또는 유니코드 수학 기호 밀도 ≥25%인 span을 `block_type="math"` 블록으로 묶고 chunker에서 분할 금지. 인라인 수식은 유니코드 그대로 텍스트 블록에 포함. **LaTeX 원본 복원은 수행하지 않음** (이미지 기반 OCR은 범위 외).

---

### L2: Figure 및 Table — 표는 해결됨, Figure는 범위 외

**표(Table)**: pymupdf4llm이 markdown 표로 변환하여 보존 + pymupdf `find_tables()` fallback. chunker에서 분할 금지. **Figure/이미지 QA는 지원하지 않음** — Vision-LM 의존 기능은 범위 외. Figure 캡션 텍스트만 검색 가능.

---

### L3: 스캔 PDF (이미지 기반) 처리 불가

**현상**: 텍스트 레이어 없는 스캔 PDF → pymupdf 빈 텍스트 반환.

**영향**: 오래된 논문, 카메라 촬영 스캔본 처리 불가.

**현재 완화**: 없음.

**향후**: PaddleOCR, Tesseract로 OCR 전처리 후 파이프라인 진입.  
슬라이드 PDF는 대부분 텍스트 레이어가 있으므로 현재도 처리 가능. 이미지 PDF(스캔본)만 OCR 필요.

---

### L4: 키워드 기반 라우팅의 취약성

**현상**: ROUTE_MAP 키워드 매칭이 비관습적 표현에서 오분류.

**예시 실패 케이스**:
- "이 식이 어떻게 도출돼?" → method 인식 실패
- "뭘 비교했어?" → compare 아닌 A경로
- "이 논문 핵심 정리" → E경로 아닌 A경로

**현재 완화**: 신뢰도 0.3 미달 시 A경로 fallback.

**향후**: `klue/roberta-small` 기반 경량 분류기 fine-tuning (쿼리-경로 200개 레이블).

---

### L5: 단일 홉 인용 추적 + arXiv 미등록 논문 한계

**현상**: Pipeline D가 직접 참고문헌만 추적, 2-hop 미지원. arXiv 미등록 논문 수집 불가.

**영향**: 지적 계보 탐색, 리뷰 논문 수준 문헌 추적 불가.

**현재 완화**: arXiv API 최대 5편 수집.

**구현 아이디어 (사용자 제안)**: "참고문헌 있으면 그걸 API로 가져와서 옆에 스페이스에 두면 해결되는 거 아닌가"
→ **실현 가능성**: 높음. 구체적 구현 방안:
1. Pipeline D에서 arXiv 수집 완료 후 PDF 링크 목록을 프론트엔드로 전달
2. 뷰어 패널(현재 PDF 뷰어 왼쪽 스페이스)에 "인용 논문 목록" 탭 추가
3. 각 항목에 "다운로드 & 인덱싱" 버튼 → arXiv 있으면 자동, 없으면 "arXiv 미등록" 표시
4. 이미 구현된 `/api/citations/download` 엔드포인트(미완성) 완성하면 됨
→ **미등록 논문**: 제목+저자+연도 메타데이터는 Semantic Scholar API로 표시 가능. PDF는 불가.

---

### L6: 대용량 코퍼스 확장성 한계

**현상**: BM25 인덱스가 메모리 내 배열, 새 논문 추가마다 전체 재구축.

**영향**: 30편 이상에서 인덱싱 속도 저하.

**현재 완화**: 배치 크기 500 (ChromaDB 제한 대응).

**향후**: BM25 증분 업데이트 + Redis-backed 공유 인덱스.

---

### L7: SCD의 기술 용어 패널티 문제

**현상**: SCD가 "BERT", "Transformer", "BLEU" 같은 영어 기술 용어에도 β 패널티 적용.

**영향**: 답변에서 모델명/지표명이 억제됨.

**현재 완화**: 숫자, 공백, 구두점만 예외.

**향후**: 논문 제목/초록에서 기술 용어 자동 추출 → 패널티 예외 whitelist 구성.

---

### L8: 멀티모달 입력 — 정책적 범위 외

**정책**: M-RAG는 텍스트 전용 RAG로 유지. YouTube/오디오/Vision-LM/이미지 QA는 모두 범위 외. 슬라이드 PDF는 pymupdf4llm 적용 후 코드/문법/표/수식 블록이 모두 텍스트 레벨에서 보존되므로 별도 멀티모달 없이 처리 가능. DOCX, TXT는 별도 파서 지원.

## 2. 기능 추가 아이디어 (사용자 제안)

### 문제지/퀴즈 생성 기능

**사용자 제안**: "이건 그냥 우리도 문제지 생성 기능 넣자, 학습으로까지 확대하면"

**구현 현실성**: 중간. 구체적 방안:
- Pipeline E(요약)에 "문제 생성" 분기 추가 (Pipeline F)
- ROUTE_MAP에 "퀴즈", "문제", "연습", "시험" 키워드 추가
- LLM 프롬프트: "아래 컨텍스트를 바탕으로 객관식 5문제를 생성하세요 (정답 포함)"
- 섹션별 문제 생성: 각 섹션(method/result/conclusion)에서 1~2문제씩

**추가 작업 범위**: 
- `config.py` ROUTE_MAP 키워드 추가: 약 5분
- `pipeline_f_quiz.py` 신규 생성 + 프롬프트 설계: 약 1~2시간
- 프론트엔드 문제지 렌더링 (markdown table): 기존 markdown 렌더링으로 커버 가능

**단점**: LLM이 문제를 "꾸며낼" 위험. CAD를 문제 생성에도 적용하면 컨텍스트 기반 문제 생성 강제 가능.

|------|-----------|-------|
| 오디오 오버뷰 (팟캐스트) | 있음 | 없음 |
| YouTube/슬라이드 입력 | 있음 | 없음 |
| 소비자 UX 완성도 | 상업 제품 수준 | 연구 프로토타입 수준 |

| 플래시카드/퀴즈 자동 생성 | 있음 | 없음 (구현 예정) |
| PPT 자동 생성 | 있음 | 없음 |

## 4. M-RAG가 학술 용도에서 더 잘 하는 것

| 기능 | M-RAG | NotebookLM |
|------|-------|-----------|
| 라우팅 투명성 | 완전 공개, 라우트 배지 표시 | 블랙박스 |

## 6. 결론

M-RAG의 가장 중요한 다음 단계는 두 가지입니다:

1. **C3 실험 실행** (CAD on/off RAGAS 비교) — 논문 방어의 핵심 증거
2. **인용 뷰어 패널 추가** (L5) — arXiv 미등록 한계를 사용자에게 명확히 보여주면서도 실제 가치 제공

arXiv 미등록/유료 논문 문제는 구조적 한계로, 완전한 해결보다 **명확한 사용자 안내**가 현실적 목표입니다.
